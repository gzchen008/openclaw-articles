# 场景模板编写指南

> **适用对象**: 需要新增或修改场景架构模板的团队成员

## 模板文件规范

### 文件命名

```
{scenario_id}-template.md
```

- `scenario_id`: 场景标识符，全小写，使用连字符分隔
- 示例: `transaction-template.md`, `query-template.md`

### 文件位置

- **基础模板库**（skill 内置）: `plugins/ease/skills/ease-arch-documentation/references/scenario-templates/`
- **项目模板库**（运行时生成）: `docs/system/architecture/templates/`

### 标准章节结构（10 章，必须完整）

每个模板 **必须** 包含以下 10 个章节，顺序固定：

```markdown
# {场景中文名}架构模板 ({场景英文名} Architecture Template)

## 模板元数据
## 1. 架构模式推荐
## 2. 技术栈推荐
## 3. 组件清单
## 4. 数据流设计
## 5. 接口契约模板
## 6. 安全考虑
## 7. 性能优化
## 8. 可观测性
## 9. 测试策略
## 10. 定制化参数
```

## 各章节编写规范

### 模板元数据

**必填项**:
- `场景类型`: 场景 ID（与文件名中的 scenario_id 一致）
- `适用用例`: 列举 3-5 个典型用例名称
- `版本`: 语义化版本（v1.0、v1.1 等）

### 1. 架构模式推荐

**必须包含**:
- **核心模式**: 1 个（最推荐的架构模式）
- **备选模式**: 1-2 个（特定条件下的替代方案）
- **简化模式**: 1 个（低复杂度场景的简化版）
- **不推荐**: 1-2 个（说明原因）

**编写原则**:
- 每个模式附带一句话说明适用条件
- 不推荐的模式必须说明具体原因

### 2. 技术栈推荐

**必须覆盖**:
- 数据库（主库 + 辅助存储）
- 缓存策略（类型 + 模式 + 注意事项）
- 消息队列（用途 + 推荐方案，标注"按需"如不必须）

**编写原则**:
- 推荐通用方案（MySQL/PostgreSQL、Redis、RabbitMQ）
- 标注与项目技术栈对齐的要求
- 特殊组件标注"按需"

### 3. 组件清单

**必须使用表格格式**:

```markdown
| 组件名 | 职责 | 必需性 |
|--------|------|--------|
| XxxService | 一句话职责描述 | 必需/推荐/按需 |
```

**必需性定义**:
- **必需**: 该场景下不可省略
- **推荐**: 建议包含，但简单场景可省略
- **按需**: 仅特定条件下需要

**编写原则**:
- 核心组件 3-5 个（必需为主）
- 扩展组件 2-4 个（推荐/按需为主）
- 组件名使用 PascalCase

### 4. 数据流设计

**必须使用 Mermaid 图**:
- 序列图（`sequenceDiagram`）: 适用于请求-响应流程
- 状态图（`stateDiagram-v2`）: 适用于状态流转
- 流程图（`flowchart`）: 适用于数据管道

**编写原则**:
- 标注核心交互路径
- 包含成功和失败两条路径
- 参与者不超过 8 个

### 5. 接口契约模板

**必须包含**:
- 核心 API 1-3 个（HTTP 方法 + 路径 + 请求/响应结构）
- 使用伪代码格式（不绑定特定框架）

**编写原则**:
- 标注必填字段
- 包含错误响应格式

### 6. 安全考虑

**必须覆盖**:
- 该场景的 Top 3 安全风险
- 每个风险的缓解措施

### 7. 性能优化

**必须使用表格格式**:

```markdown
| 指标 | 目标 | 优化策略 |
|------|------|---------|
| 具体指标 | 量化目标值 | 1-2 条优化措施 |
```

**编写原则**:
- 指标必须可量化（TPS、延迟 P99 等）
- 目标值给出参考范围而非绝对值

### 8. 可观测性

**必须包含**:
- 关键指标（3-5 个）
- 告警阈值（2-3 个）

### 9. 测试策略

**必须使用表格格式**:

```markdown
| 测试类型 | 重点场景 |
|----------|---------|
| 单元/集成/性能/安全 | 该类型下的 2-3 个关键场景 |
```

### 10. 定制化参数

**必须使用表格格式**:

```markdown
| 参数名 | 说明 | 默认值 |
|--------|------|--------|
| UPPER_SNAKE_CASE | 一句话说明 | 具体默认值 |
```

**编写原则**:
- 参数名使用 `UPPER_SNAKE_CASE`
- 提供合理的默认值
- 参数数量 3-8 个

## 定制化策略

### 技术栈对齐

当模板推荐的技术栈与项目现有技术栈不同时：

1. **优先使用项目现有技术栈**（降低运维复杂度）
2. **在架构设计文档中记录调整**:
   ```markdown
   #### 调整: 数据库
   - **模板推荐**: MySQL
   - **项目现有**: PostgreSQL
   - **调整决策**: 使用 PostgreSQL
   - **理由**: 系统已有 PostgreSQL 运维经验
   ```

### 组件裁剪

根据用例复杂度裁剪组件：

1. **简单用例**（单服务、低并发）: 仅保留"必需"组件
2. **中等用例**（跨模块、中并发）: 保留"必需" + "推荐"组件
3. **复杂用例**（跨服务、高并发）: 使用完整组件列表

### 架构模式调整

根据实际场景调整架构模式：

1. **单服务场景**: 使用"简化模式"
2. **跨服务场景**: 使用"核心模式"
3. **高性能场景**: 根据性能要求选择最优模式

## 新增模板流程

1. 在 `scenario-templates/` 目录下创建 `{scenario_id}-template.md`
2. 按标准 10 章结构编写内容
3. 在 `scenario-classification.md` 中添加新场景的关键词库和识别规则
4. 更新 `scenario-templates/README.md` 索引
5. 更新 `generate-scenario-templates.md` 中的场景列表

## 质量检查清单

- [ ] 文件名符合 `{scenario_id}-template.md` 格式
- [ ] 包含完整的 10 个标准章节
- [ ] 模板元数据中的场景类型与文件名一致
- [ ] 架构模式推荐包含核心/备选/简化/不推荐四项
- [ ] 组件清单使用表格格式，标注必需性
- [ ] 数据流使用 Mermaid 图
- [ ] 性能优化指标可量化
- [ ] 定制化参数有合理默认值
- [ ] 已更新 README.md 索引
- [ ] 已更新 scenario-classification.md
