# install-update-ease-plugin

本 skill 用于规范 Ease 插件的离线安装与更新自动化流程。默认由 Claude 直接代为执行完整闭环：临时 clone 仓库、将该仓库注册为本地 marketplace、安装或刷新 `ease` 插件、验证结果、清理临时目录。除非遇到真实阻塞条件，否则不要把命令转交给用户手动执行。仓库选择必须遵循固定优先级：先使用 `inf` 组仓库，若因权限不足或仓库不可访问而失败，再回退使用 `rcp` 组仓库。无论是首次安装还是后续更新，均不依赖用户长期保留本地 clone。

## 适用场景

**在以下场景使用：**

- 用户希望通过离线方式安装 Ease 插件
- 用户需要刷新或更新已安装的 Ease 插件
- 用户所在环境无法直接通过在线 marketplace 安装
- 用户明确要求使用内网仓库 + 离线 marketplace 的方式安装
- 用户希望 Claude 全自动完成安装或更新，不自己执行命令

**以下场景不要使用：**

- 用户只是想了解 Ease 插件是什么
- 用户要修改插件源码
- 用户已有明确的其他安装规范

## 执行原则

- 默认由 Claude 直接执行安装、更新、验证与清理，不输出待用户手动执行的命令清单
- 仓库优先级固定为：先 `git@code.weoa.com:inf/claude-skills-marketplace.git`，失败后再 `git@code.weoa.com:rcp/claude-skills-marketplace.git`
- 本地安装方式必须通过“添加本地 marketplace → 安装 `ease` 插件”完成，不能把本地目录直接作为 `plugin install` 的参数
- 用于注册 marketplace 的路径必须是临时 clone 出来的仓库根目录；`plugins/ease` 仍然是仓库内实际插件位置，不可误解为安装命令直接接收的路径参数
- 安装与更新都必须使用临时 clone 目录，不依赖用户长期保留的本地仓库
- 成功标准是插件完成安装或刷新且验证通过；验证失败视为未完成
- 无论成功或失败，原则上都应清理临时 clone 目录；若清理失败，必须向用户说明残留路径
- 普通本地安装、更新、验证、临时目录删除属于默认自动执行范围，无需用户手动介入
- 只有遇到权限、网络、环境限制等 Claude 无法自行消除的阻塞时，才向用户说明唯一需要配合的事项

## 仓库优先级

按以下顺序尝试：

1. `git@code.weoa.com:inf/claude-skills-marketplace.git`
2. `git@code.weoa.com:rcp/claude-skills-marketplace.git`

如果 `inf` 仓库因无权限、仓库不可访问或同类阻塞原因 clone 失败，应立即回退到 `rcp` 仓库继续执行，而不是停止在失败状态等待用户决定。

## 标准安装流程

首次安装时，Claude 应执行以下自动化闭环：

1. 创建临时目录，并优先从 `inf` 仓库 clone。
2. 若 `inf` clone 失败且原因为权限或访问受限，则自动回退到 `rcp` 仓库重新 clone。
3. 将临时 clone 出来的仓库根目录添加为本地 marketplace。
4. 通过本地 marketplace 安装 `ease` 插件，而不是把本地目录直接传给 `plugin install`。
5. 自动验证安装结果，包括插件是否出现在插件列表中、Ease 相关命令是否可正常使用。
6. 验证通过后向用户汇报安装成功。
7. 无论安装成功或失败，原则上都删除本次临时 clone 的目录；若删除失败，需明确告知残留路径。

如需描述执行动作，可将命令视为 Claude 的内部执行步骤，而不是给用户的教程。关键要求是：先把临时仓库根目录注册为本地 marketplace，再安装 `ease` 插件；安装完成后必须验证并清理临时目录。

## 标准更新流程

更新时，Claude 应与安装保持同一自动化闭环，不依赖用户本地长期存在的 clone：

1. 创建新的临时目录，并优先从 `inf` 仓库 clone 最新内容。
2. 若 `inf` clone 失败且原因为权限或访问受限，则自动回退到 `rcp` 仓库重新 clone。
3. 将临时 clone 出来的仓库根目录添加为本地 marketplace。
4. 通过本地 marketplace 重新安装或刷新 `ease` 插件，以更新已安装插件。
5. 自动验证更新结果，包括插件可见性与 Ease 命令可用性。
6. 验证通过后向用户汇报更新成功。
7. 无论更新成功或失败，原则上都删除本次临时 clone 的目录；若删除失败，需明确告知残留路径。

不要把 `git -C ... pull --ff-only` 作为默认更新路径，也不要要求用户长期保留某个本地仓库作为更新前提。更新的默认策略始终是：临时 clone 最新仓库 → 添加本地 marketplace → 刷新 `ease` → 验证 → 删除临时目录。

## 跨平台命令注意事项

- 涉及命令行时，必须区分 Linux/macOS 与 Windows，不能把单一平台的行为当作通用规则。
- Claude 在内部执行命令时，应优先使用当前环境原生命令形式，并避免把某个平台专属写法直接复用到另一平台。
- 在 Windows 尤其是 Git Bash 场景下，不要假设 `/plugin ...` 与 Linux/macOS 中表现一致；应优先使用 `claude plugin ...` 形式，避免被路径转换机制误解析。
- 在 Linux/macOS 场景下，也应优先采用 `claude plugin ...` 这类明确的 CLI 形式，以减少 shell 差异带来的歧义。
- 若需要向用户解释执行方式，应明确说明平台前提，例如“这是 Windows Git Bash 特有问题”或“这是类 Unix shell 下的行为”。
- 如果某个平台存在命令兼容性问题，默认策略不是让用户自己猜命令，而是由 Claude 选择该平台可用的等价执行方式。

## 故障排查

| 现象 | 可能原因 | 处理方式 |
| --- | --- | --- |
| clone `inf` 仓库时报 `Permission denied` | 当前账号无 `inf` 组仓库权限 | 自动改用 `rcp` 仓库继续执行 |
| `inf` 与 `rcp` 都无法访问 | 当前环境无法连接目标 Git 仓库 | 向用户说明当前唯一阻塞点是仓库访问或权限问题 |
| 直接把本地路径传给 `plugin install` 后安装失败 | CLI 只会从已配置 marketplace 中查找插件，不支持把本地插件目录直接作为安装参数 | 改为先把临时 clone 的仓库根目录添加为本地 marketplace，再安装 `ease` |
| 在 Git Bash 中执行 `/plugin ...` 行为异常 | Windows Git Bash 把 `/plugin` 误解析为本地文件系统路径 | 改用 `claude plugin ...` 形式，由 Claude 自动执行，不向用户下发 `/plugin` 风格命令 |
| 安装或更新后看不到命令 | 安装未成功或验证未通过 | 重新核对插件列表与 Ease 命令可用性，未通过则视为未完成 |
| 临时目录删除失败 | 文件占用或权限限制 | 向用户说明清理失败，并给出残留路径 |

## 对用户输出规范

对用户的输出应是结果导向、代执行导向，而不是命令教程。推荐表达方式：

- 我将为你自动安装或更新 Ease 插件。
- 我会优先尝试 `inf` 仓库，如无权限会自动回退到 `rcp` 仓库。
- 我会使用临时目录完成 clone、添加本地 marketplace、安装或刷新 `ease`、验证，并在结束后清理临时目录。
- 如果遇到我无法自行消除的权限、网络或环境阻塞，我会明确告诉你唯一需要配合的事项。
- 如需说明命令行为差异，我会明确区分 Windows 与 Linux/macOS，而不会把单一平台命令当成通用方案。
- 完成后我会直接汇报结果；如果清理未完成，我会说明残留路径。

禁止默认输出按步骤让用户手动执行的命令清单。即使为了说明内部动作而提到命令，也必须明确那是 Claude 的执行动作，不是要求用户自己操作。

## 常见错误

- 明明要求离线安装，却仍建议使用在线 marketplace
- 把 skill 写成教用户手动执行命令，而不是 Claude 自动代执行
- 把本地插件目录直接传给 `plugin install`，而不是先添加本地 marketplace 再安装 `ease`
- 混淆仓库根目录与 `plugins/ease` 的角色：前者用于注册本地 marketplace，后者是仓库内实际插件位置
- 遇到 `inf` 无权限时，没有自动回退到 `rcp`
- 更新流程仍依赖长期保留的本地 clone，或默认使用 `git -C ... pull --ff-only`
- 安装或更新完成后未验证就宣布成功
- 安装或更新完成后忘记删除临时 clone 仓库，给用户增加额外磁盘占用
