# 任务： [FEATURE NAME]

**输入**：来自 `.eases/[###-feature-name]/[command]/` 的设计文档  
**前置条件**：plan.md（必需）、spec.md（用于用户故事，必需）、research.md、data-model.md、contracts/

**测试**：下方示例包含测试相关任务。测试为可选项——仅在功能规范明确请求时才包含。

**组织方式**：任务按用户故事分组，以便每个故事可独立实施与测试。

## 格式：`[ID] [P?] [Story] 描述`

- **[P]**：可并行执行（不同文件、无依赖）
- **[Story]**：该任务所属的用户故事（例如：US1、US2、US3）
- 在描述中包含精确的文件路径

## 路径约定

- **单项目**：仓库根目录下使用 `src/`、`tests/`
- **Web 应用**：使用 `backend/src/`、`frontend/src/`
- **移动端**：使用 `api/src/`、`ios/src/` 或 `android/src/`
- 下方路径示例默认按单项目结构给出——请依据 plan.md 的结构进行调整

## 依赖与执行顺序

### 阶段依赖

- **搭建（第 1 阶段）**：无依赖——可立即开始
- **基础（第 2 阶段）**：依赖搭建完成——阻塞所有用户故事
- **用户故事（第 3 阶段及以后）**：均依赖基础阶段完成
  - 随后用户故事可并行推进（若人力允许）
  - 或按优先级顺序串行推进（P1 → P2 → P3）
- **打磨（最终阶段）**：依赖所有目标用户故事完成

### 用户故事依赖

- **用户故事 1（P1）**：基础阶段完成后可开始——不依赖其他故事
- **用户故事 2（P2）**：基础阶段完成后可开始——可与 US1 集成，但须可独立测试
- **用户故事 3（P3）**：基础阶段完成后可开始——可与 US1/US2 集成，但须可独立测试

### 每个用户故事内部

- 如包含测试，必须先编写并确保在实现前失败
- 先模型，后服务
- 先服务，后端点
- 先核心实现，后集成
- 每个故事完成后再进入下一优先级

### 并行机会

- 所有标记为 [P] 的“搭建”任务可并行
- 所有标记为 [P] 的“基础”任务可并行（第 2 阶段内部）
- 基础阶段完成后，所有用户故事可并行（视团队产能）
- 某用户故事的所有测试中标记为 [P] 的任务可并行
- 某故事内的模型任务如标记为 [P] 可并行
- 不同用户故事可由不同成员并行推进

## 实施策略

### 先交付 MVP（仅用户故事 1）

1. 完成第 1 阶段：搭建
2. 完成第 2 阶段：基础（关键——阻塞所有故事）
3. 完成第 3 阶段：用户故事 1
4. 停止并验证：独立测试用户故事 1
5. 如准备就绪，部署/演示

### 增量式交付

1. 完成“搭建 + 基础”→ 基础就绪
2. 添加用户故事 1 → 独立测试 → 部署/演示（MVP！）
3. 添加用户故事 2 → 独立测试 → 部署/演示
4. 添加用户故事 3 → 独立测试 → 部署/演示
5. 每个故事都在不破坏先前故事的前提下产生增量价值

### 并行团队策略

多人协作时：

1. 团队共同完成“搭建 + 基础”
2. 基础完成后：
   - 开发者 A：用户故事 1
   - 开发者 B：用户故事 2
   - 开发者 C：用户故事 3
3. 各故事独立完成与集成

## 备注

- [P] 任务 = 不同文件、无依赖
- [Story] 标签将任务与具体用户故事关联，便于追踪
- 每个用户故事应可独立完成与测试
- 在实现前先确保测试失败
- 每个任务或逻辑组完成后进行提交
- 在任何检查点暂停以独立验证故事
- 避免：含糊任务、同一文件冲突、破坏独立性的跨故事依赖
