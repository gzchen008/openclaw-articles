## 清单目标："英文需求的单元测试"

关键理念：清单是“需求写作的单元测试”——它们用于验证在特定领域中需求的质量、清晰度与完整性。

配合ease-spec技能完成任务，其中scripts/bash/create-new-feature.sh等脚本位于ease-spec技能包中。

不用于验证/测试实现：

- ❌ 不是“验证按钮点击是否正确”
- ❌ 不是“测试错误处理是否工作”
- ❌ 不是“确认 API 返回 200”
- ❌ 不是检查代码/实现是否符合规范

用于需求质量验证：

- ✅ “是否为所有卡片类型定义了视觉层级需求？”（完整性）
- ✅ “是否为‘突出显示’提供了具体尺寸/位置的量化标准？”（清晰度）
- ✅ “所有交互元素的悬停状态需求是否一致？”（一致性）
- ✅ “是否为键盘导航定义了可访问性需求？”（覆盖度）
- ✅ “规范是否定义了当 Logo 图片加载失败时的处理行为？”（边界情况）

比喻：如果你的规范是用英文写的“代码”，那么清单就是它的单元测试套件。你在测试的是需求是否写得好、是否完整、是否不含歧义并准备好实施——而不是测试实现是否工作。

## 用户输入

```text
$ARGUMENTS
```

在继续之前（如不为空），你必须考虑用户输入。

## 执行步骤

1. 设定：从仓库根目录运行 `{SCRIPT}` 并解析 JSON 以获取 FEATURE_DIR 与 AVAILABLE_DOCS 列表。

   - 所有文件路径必须为绝对路径。
   - 对于类似 "I'm Groot" 的单引号参数，使用转义语法：例如 'I'\''m Groot'（或在可能情况下使用双引号："I'm Groot"）。
2. 澄清意图（动态）：派生最多三个初始上下文澄清问题（不使用预设问题库）。它们必须：

   - 来自用户措辞与从 spec/plan/tasks 中提取的信号
   - 仅询问会实质性影响清单内容的信息
   - 若某点在 `$ARGUMENTS` 中已不含歧义，则单个问题应被跳过
   - 相比广度，更偏好精确性

   生成算法：

   1. 提取信号：特性领域关键词（如 auth、latency、UX、API）、风险指示词（“critical”、“must”、“compliance”）、利益相关者线索（“QA”、“review”、“security team”）、明确交付物（“a11y”、“rollback”、“contracts”）。
   2. 将信号聚类为候选关注领域（最多 4 个），按相关性排序。
   3. 识别可能的受众与时间（作者、评审者、QA、发布），若未明确则推断。
   4. 检测缺失维度：范围广度、深度/严谨度、风险强调、排除边界、可度量的验收标准。
   5. 从下列原型中制定问题：
      - 范围精炼（如：“是否应包含与 X 和 Y 的集成触点，还是仅限于本地模块正确性？”）
      - 风险优先级（如：“这些潜在风险领域中哪些应作为强制门控检查？”）
      - 深度校准（如：“这是轻量的提交前自检清单还是正式的发布门槛？”）
      - 受众框定（如：“此清单仅供作者使用，还是用于 PR 同侪评审？”）
      - 边界排除（如：“本轮是否明确排除性能调优项？”）
      - 场景类别缺口（如：“未检测到恢复流程——回滚/部分失败路径是否纳入范围？”）

   问题格式规则：

   - 若提供选项，生成紧凑表格，列为：Option | Candidate | Why It Matters
   - 选项最多 A–E；若自由回答更合适，则省略表格
   - 切勿要求用户重复他们已说过的内容
   - 避免臆测类别（不凭空想象）。若不确定，明确询问：“请确认 X 是否属范围内。”

   当无法互动时的默认值：

   - 深度：标准
   - 受众：若涉代码则为评审者（PR），否则为作者
   - 关注：取前两名相关性最高的聚类

   输出问题（标签 Q1/Q2/Q3）。在获得回答后：若至少两个场景类别（备用/异常/恢复/非功能领域）仍不明确，你可以再问最多两个更有针对性的后续问题（Q4/Q5），每个附带一句理由（如：“恢复路径风险未解”）。总计不超过 5 个问题。若用户明确拒绝更多问题，应停止升级。
3. 理解用户请求：结合 `$ARGUMENTS` 与澄清回答：

   - 提炼清单主题（如 security、review、deploy、ux）
   - 汇总用户明确提出的必备项
   - 将关注选择映射到类别脚手架
   - 从 spec/plan/tasks 中推断缺失上下文（不得臆测）
4. 加载特性上下文：从 FEATURE_DIR 读取：

   - spec.md：特性需求与范围
   - plan.md（如存在）：技术细节与依赖
   - tasks.md（如存在）：实施任务

   上下文加载策略：

   - 仅加载与当前关注领域相关的必要部分（避免整篇灌入）
   - 对长段落优先做简明场景/需求要点总结
   - 使用渐进披露：仅在检测到缺口时增加后续检索
   - 若源文档很大，优先生成中期摘要项，而不是嵌入原文
5. 生成清单——创建“英文需求的单元测试”：

   - 若不存在则创建 `FEATURE_DIR/checklists/` 目录
   - 生成唯一清单文件名：
     - 使用基于领域的简短描述名（如 `ux.md`、`api.md`、`security.md`）
     - 格式：`[domain].md`
     - 若文件已存在，则向现有文件追加
   - 项目按顺序编号，从 CHK001 开始
   - 每次质量检查清单生成（参考reference/质量检查清单.md）运行都会创建一个新文件（切勿覆盖现有清单）

   核心原则——测试的是需求，而非实现：
   每条清单项必须从以下维度评估需求本身：

   - 完整性：是否涵盖所有必要需求？
   - 清晰度：需求是否无歧义且具体？
   - 一致性：需求之间是否协调一致？
   - 可度量性：是否能客观验证？
   - 覆盖度：是否涵盖所有场景/边界情况？

   类别结构——按需求质量维度分组：

   - 需求完整性（是否记录了所有必要需求？）
   - 需求清晰度（需求是否具体且不含歧义？）
   - 需求一致性（是否相互协调且无冲突？）
   - 验收标准质量（成功标准是否可度量？）
   - 场景覆盖（是否涵盖全部流程/案例？）
   - 边界情况覆盖（是否定义边界条件？）
   - 非功能性需求（性能、安全、可访问性等——是否已明确？）
   - 依赖与假设（是否记录并验证？）
   - 歧义与冲突（哪些需要澄清？）

   如何编写清单项——“英文的单元测试”：

   错误示例（测试实现）：

   - “验证着陆页展示 3 张剧集卡片”
   - “测试桌面端悬停状态是否工作”
   - “确认点击 Logo 导航到首页”

   正确示例（测试需求质量）：

   - “是否明确规定了精选剧集的数量与布局？”【完整性】
   - “‘突出显示’是否用具体尺寸/位置进行了量化？”【清晰度】
   - “所有交互 UI 的悬停状态需求是否一致？”【一致性】
   - “是否为所有交互 UI 定义了键盘导航需求？”【覆盖度】
   - “当 Logo 图片加载失败时，是否指定了回退行为？”【边界情况】

   项目结构：
   每条项目应遵循以下模式：

   - 以问题形式关注需求质量
   - 聚焦于规范/计划中“是否写了”（或未写）
   - 在尾部加质量维度标签【完整性/清晰度/一致性/等】
   - 检查既有需求时引用规范章节 `[Spec §X.Y]`
   - 检查缺失需求时使用 `[Gap]` 标记

   按质量维度的示例：

   完整性：

   - “是否为所有 API 失败模式定义了错误处理需求？【Gap】”
   - “是否为所有交互元素指定了可访问性需求？【完整性】”
   - “是否为响应式布局定义了移动断点需求？【Gap】”

   清晰度：

   - “‘快速加载’是否用具体时间阈值量化？【清晰度，Spec §NFR-2】”
   - “‘相关剧集’的选择标准是否明确定义？【清晰度，Spec §FR-5】”
   - “‘突出’是否用可度量的视觉属性定义？【歧义，Spec §FR-4】”

   一致性：

   - “导航需求在所有页面之间是否一致？【一致性，Spec §FR-10】”
   - “卡片组件需求在着陆页与详情页之间是否一致？【一致性】”

   覆盖度：

   - “是否为零状态场景（无剧集）定义了需求？【覆盖度，边界情况】”
   - “是否涵盖并发用户交互场景？【覆盖度，Gap】”
   - “是否为部分数据加载失败定义了需求？【覆盖度，异常流程】”

   可度量性：

   - “视觉层级需求是否可衡量/可测试？【验收标准，Spec §FR-1】”
   - “‘平衡的视觉权重’是否可客观验证？【可度量性，Spec §FR-2】”

   场景分类与覆盖（聚焦需求质量）：

   - 检查是否存在：主流程、备用流程、异常/错误、恢复、非功能性场景的需求
   - 对每类场景询问：“该场景的需求是否完整、清晰且一致？”
   - 若场景类别缺失：“该场景类型的需求是有意排除还是缺失？【Gap】”
   - 对状态变更类场景包含韧性/回滚：“是否为迁移失败定义了回滚需求？【Gap】”

   可追溯性需求：

   - 最低要求：≥80% 的清单项必须包含至少一个可追溯引用
   - 每条项目应引用：规范章节 `[Spec §X.Y]`，或使用标记：`[Gap]`、`[Ambiguity]`、`[Conflict]`、`[Assumption]`
   - 若不存在 ID 体系：“是否建立了需求与验收标准的 ID 方案？【Traceability】”

   暴露并解决问题（需求质量问题）：
   针对需求本身提出问题：

   - 歧义：“‘快速’是否用具体指标量化？【Ambiguity，Spec §NFR-1】”
   - 冲突：“§FR-10 与 §FR-10a 的导航需求是否冲突？【Conflict】”
   - 假设：“‘播客 API 始终可用’的假设是否得到验证？【Assumption】”
   - 依赖：“外部播客 API 的需求是否已记录？【Dependency，Gap】”
   - 缺少定义：“‘视觉层级’是否用可度量标准进行了定义？【Gap】”

   内容整合：

   - 软上限：若原始候选项 > 40，按风险/影响优先
   - 合并检查同一需求方面的近重复项
   - 若低影响的边界情况 > 5，创建一条合并项：“是否在需求中覆盖了边界情况 X、Y、Z？【Coverage】”

   🚫 严禁——这些会把它变成实现测试，而不是需求测试：

   - ❌ 以“Verify/Test/Confirm/Check + 实现行为”起头的任何项目
   - ❌ 引用代码执行、用户操作、系统行为
   - ❌ “显示正确”、“工作正常”、“函数如预期”
   - ❌ “点击”、“导航”、“渲染”、“加载”、“执行”
   - ❌ 测试用例、测试计划、QA 程序
   - ❌ 实现细节（框架、API、算法）

   ✅ 必需模式——这些测试的是需求质量：

   - ✅ “是否为[场景]定义/指定/记录了[需求类型]？”
   - ✅ “‘模糊术语’是否以具体标准量化/澄清？”
   - ✅ “在[章节 A]与[章节 B]之间，[需求]是否一致？”
   - ✅ “[需求]是否可客观度量/验证？”
   - ✅ “是否在需求中覆盖了[边界情况/场景]？”
   - ✅ “规范是否定义了[缺失方面]？”
6. 结构参考：生成清单时，遵循 `templates/docs/checklist-template.md` 的规范模板，包括标题、元信息区、类别标题与 ID 格式。若模板不可用，则使用：一级标题、目的/创建元信息行、`##` 类别分节，每节中包含以全局递增 ID（从 CHK001 起）标记的 `- [ ] CHK### <需求项目>` 行。
7. 报告：输出创建的清单完整路径、项目数量，并提醒用户每次运行都会创建新文件。总结：

   - 所选关注领域
   - 深度级别
   - 参与者/时间
   - 已纳入的用户明确必备项

重要说明：每次质量检查清单生成（参考reference/质量检查清单.md）调用都会创建一个使用简短描述名的清单文件，除非该文件已存在。这允许：

- 针对不同类型创建多个清单（例如 `ux.md`、`test.md`、`security.md`）
- 使用简单、易记的文件名来指示清单用途
- 在 `checklists/` 文件夹中易于识别与导航

为避免杂乱，请使用描述性类型，并在完成后清理过时的清单。

## 反例：不应做的事

错误示例——这些测试的是实现，而非需求：

```markdown
- [ ] CHK001 - 验证着陆页展示 3 张剧集卡片 [Spec §FR-001]
- [ ] CHK002 - 测试桌面端悬停状态工作正常 [Spec §FR-003]
- [ ] CHK003 - 确认点击 Logo 导航到首页 [Spec §FR-010]
- [ ] CHK004 - 检查“相关剧集”区显示 3–5 项 [Spec §FR-005]
```

正确示例——这些测试的是需求质量：

```markdown
- [ ] CHK001 - 是否明确规定了精选剧集的数量与布局？【完整性，Spec §FR-001】
- [ ] CHK002 - 所有交互元素的悬停状态需求是否一致？【一致性，Spec §FR-003】
- [ ] CHK003 - 品牌元素的导航需求是否清晰？【清晰度，Spec §FR-010】
- [ ] CHK004 - “相关剧集”的选择标准是否已记录？【Gap，Spec §FR-005】
- [ ] CHK005 - 是否为异步剧集数据定义了加载状态需求？【Gap】
- [ ] CHK006 - “视觉层级”需求是否可客观度量？【可度量性，Spec §FR-001】
```

关键差异：

- 错误：测试系统是否工作正确
- 正确：测试需求是否写得正确
- 错误：验证行为
- 正确：验证需求质量
- 错误：“是否做到 X？”
- 正确：“是否清晰地规定了 X？”
