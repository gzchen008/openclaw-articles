## 用户输入

```text
$ARGUMENTS
```

在继续之前，若用户输入不为空，必须优先考虑该输入。

## 纲要

配合ease-spec技能完成任务，其中scripts/bash/create-new-feature.sh等脚本位于ease-spec技能包中。

当用户请求创建需求规范时，提供的功能描述即为输入。即使下方出现字面量 `{ARGS}`，也假定在本会话中始终可获取该描述。除非用户命令为空，不要要求用户重复描述。

给定该功能描述，执行以下步骤：

1. 生成分支的简短名称（2-4 个词）

   - 分析功能描述，提炼最有意义的关键词
   - 创建能抓住特性本质的 2-4 词短名称
   - 尽量使用“动作-名词”格式，例如 “add-user-auth”、"fix-payment-bug"
   - 保留技术术语与缩写（OAuth2、API、JWT 等）
   - 保持简洁，并保证可一眼理解该特性
   - 示例：
     - “我想添加用户认证功能” → “user-auth”
     - “为 API 实现 OAuth2 集成” → “oauth2-api-integration”
     - “创建分析看板” → “analytics-dashboard”
     - “修复支付处理超时问题” → “fix-payment-timeout”
2. 在创建新分支前检查是否已有分支：

   a. 先获取所有远程分支，确保信息最新：

   ```bash
   git fetch --all --prune
   ```

   b. 在所有来源中为该短名称查找最高特性编号：

   - 远程分支：`git ls-remote --heads origin | grep -E 'refs/heads/[0-9]+-<short-name>$'`
   - 本地分支：`git branch | grep -E '^[* ]*[0-9]+-<short-name>$'`
   - 规范目录：检查匹配 `features/[0-9]+-<short-name>` 的目录

   c. 确定下一个可用编号：

   - 从三类来源中提取所有编号
   - 找到最高编号 N
   - 新分支编号使用 N+1

   d. 使用计算得出的编号与短名称运行脚本 `{SCRIPT}`：

   - 传入 `--number N+1` 与 `--short-name "your-short-name"`，连同功能描述
   - Bash 示例：`{SCRIPT} --json --number 5 --short-name "user-auth" "Add user authentication"`
   - PowerShell 示例：`{SCRIPT} -Json -Number 5 -ShortName "user-auth" "Add user authentication"`

   重要事项：

   - 同时检查三类来源（远程分支、本地分支、规范目录）以找到最高编号
   - 仅匹配与该短名称完全一致的分支/目录模式
   - 若未发现与该短名称匹配的分支/目录，则从编号 1 开始
   - 每个功能仅能运行该脚本一次
   - 终端会输出 JSON——始终参考该输出以获取所需的实际内容
   - JSON 输出将包含 BRANCH_NAME 与 SPEC_FILE 路径
   - 当参数中包含单引号（如 "I'm Groot"），使用转义语法：例如 `'I'\''m Groot'`（或尽量使用双引号 `"I'm Groot"`）
3. 加载ease-spec技能中的 `templates/spec-template.md`，理解所需章节与结构。
4. 按以下执行流程进行：

   1. 解析输入中的用户描述若为空：错误 "No feature description provided"
   2. 从描述中抽取关键概念识别：参与者、行为、数据、约束
   3. 对不明确之处：
      - 在上下文与行业惯例基础上做出合理假设
      - 仅当下列情况使用 [NEEDS CLARIFICATION: 具体问题] 标记：
        - 该选择会显著影响特性范围或用户体验
        - 存在多种合理解释且影响不同
        - 不存在合理默认值
      - 限制：最多 3 个 [NEEDS CLARIFICATION] 标记
      - 按影响优先级排序：范围 > 安全/隐私 > 用户体验 > 技术细节
   4. 填写“用户场景与测试”章节若无法明确用户流程：错误 "Cannot determine user scenarios"
   5. 生成功能性需求每项需求必须可测试对未指明的细节使用合理默认（并在“假设”章节记录）
   6. 定义成功标准创建可度量、与技术无关的结果同时包含定量指标（时间、性能、数量）与定性指标（用户满意度、任务完成）每条标准均应可在不依赖实现细节的前提下验证
   7. 识别关键实体（如涉及数据）
   8. 返回：SUCCESS（规范可进入规划阶段）
5. 使用模板结构将规范写入 SPEC_FILE，替换占位符为基于功能描述（参数）提炼出的具体细节，同时保持章节顺序与标题不变。
6. 规范质量校验：写入初版规范后，依据质量标准进行校验：

   a. 创建“规范质量清单”：在 `.eases/FEATURE_DIR/checklists/requirements.md` 生成一个清单文件，采用清单模板结构，并包含以下校验项：

   ```markdown
   # 规范质量清单： [FEATURE NAME]

   **目的**：在进入规划前验证规范的完整性与质量  
   **创建时间**： [DATE]  
   **功能**： [链接至 spec.md]

   ## 内容质量

   - [ ] 不包含实现细节（语言、框架、API）
   - [ ] 聚焦用户价值与业务需求
   - [ ] 面向非技术型干系人撰写
   - [ ] 所有必填章节均已完成

   ## 需求完整性

   - [ ] 不包含 [NEEDS CLARIFICATION] 标记
   - [ ] 需求可测试且无歧义
   - [ ] 成功标准可度量
   - [ ] 成功标准与技术无关（不含实现细节）
   - [ ] 已定义所有验收场景
   - [ ] 已识别边界情况
   - [ ] 范围界定清晰
   - [ ] 已识别依赖与假设

   ## 特性就绪度

   - [ ] 所有功能性需求均具备明确的验收标准
   - [ ] 用户场景覆盖主要流程
   - [ ] 特性满足“成功标准”中定义的可度量结果
   - [ ] 规范未泄露任何实现细节

   ## 备注

   - 标记为未完成的项目，须在进入澄清阶段（参考reference/需求澄清.md）或计划阶段（参考reference/制定技术计划.md）前更新规范
   ```

   b. 运行校验检查：逐条审阅规范与每个清单项

   - 对每项判断“通过/失败”
   - 记录发现的具体问题（引用相关规范段落）

   c. 处理校验结果：

   - 若所有项通过：将清单标记为完成并进入第 6 步
   - 若存在失败项（不含 [NEEDS CLARIFICATION]）：

     1. 列出失败项与具体问题
     2. 更新规范以修复问题
     3. 重新运行校验，直到所有项通过（最多 3 轮）
     4. 若 3 轮后仍失败，将剩余问题记录在清单备注并向用户告警
   - 若仍存在 [NEEDS CLARIFICATION] 标记：

     1. 从规范中提取所有 [NEEDS CLARIFICATION: ...] 标记
     2. 限制检查：如超过 3 个，仅保留影响最关键的 3 个（按范围/安全/UX 影响），其余做合理假设
     3. 针对每个澄清项（最多 3 个），按以下格式向用户呈现选项：

        ```markdown
        ## Question [N]: [Topic]

        **Context**: [引用相关规范段落]

        **What we need to know**: [来自 NEEDS CLARIFICATION 标记的具体问题]

        **Suggested Answers**:

        | Option | Answer | Implications |
        |--------|--------|--------------|
        | A      | [第一种建议答案] | [对特性的影响] |
        | B      | [第二种建议答案] | [对特性的影响] |
        | C      | [第三种建议答案] | [对特性的影响] |
        | Custom | 请提供自定义答案 | [说明如何提供自定义输入] |

        **Your choice**: _[等待用户回复]_
        ```
     4. 关键——表格格式：确保 Markdown 表格格式正确

        - 使用一致的竖线与对齐
        - 每个单元两侧留空格：`| 内容 |` 而非 `|内容|`
        - 表头分隔符至少包含 3 个短横线：`|--------|`
        - 在 Markdown 预览中验证渲染效果
     5. 顺序编号问题（Q1、Q2、Q3——最多 3 个）
     6. 在等待回复前一次性呈现所有问题
     7. 用户回复选择后（如 “Q1: A, Q2: Custom - [详情], Q3: B”），将规范中的对应 [NEEDS CLARIFICATION] 替换为所选答案
     8. 所有澄清解决后重新运行校验

   d. 更新清单：每次校验迭代后，更新清单文件中的通过/失败状态
7. 汇报完成：包含分支名称、规范文件路径、清单结果，以及准备进入下一阶段（澄清阶段参考reference/需求澄清.md 或计划阶段参考reference/制定技术计划.md）。

注意：脚本会创建并检出新分支，并在写入规范前初始化规范文件。

## 通用指南

## 快速指南

- 聚焦用户需要“是什么”与“为什么”。
- 避免“如何实现”（不涉及技术栈、API、代码结构）。
- 面向业务干系人撰写，而非开发者。
- 不要在规范中内嵌任何清单。清单由单独命令生成。

### 章节要求

- 必填章节：每个功能都必须完成
- 可选章节：仅在相关时包含
- 如某章节不适用，直接删除（不要保留 “N/A”）

### AI 生成指南

根据用户提示创建规范时：

1. 做出合理假设：依据上下文、行业标准与常见模式补足空缺
2. 记录假设：在“假设”章节中记录合理默认
3. 限制澄清：最多 3 个 [NEEDS CLARIFICATION]——仅用于如下关键决策：
   - 显著影响特性范围或用户体验
   - 存在多种合理解释且影响不同
   - 不存在任何合理默认
4. 澄清优先级：范围 > 安全/隐私 > 用户体验 > 技术细节
5. 以测试者视角思考：含糊需求应无法通过“可测试且无歧义”的清单项
6. 常见需澄清领域（仅在无合理默认时提出）：
   - 特性范围与边界（明确包含/排除的用例）
   - 用户类型与权限（存在多种冲突解释时）
   - 安全/合规要求（涉及法律/财务重要性）

合理默认示例（无需询问）：

- 数据保留：使用所在领域的行业惯例
- 性能目标：采用标准 Web/移动应用预期，除非另有说明
- 错误处理：友好消息与适当回退
- 认证方式：Web 应用采用标准会话或 OAuth2
- 集成模式：未说明则默认 RESTful API

### 成功标准指南

成功标准必须：

1. 可度量：包含具体指标（时间、百分比、数量、速率）
2. 与技术无关：不提及框架、语言、数据库或工具
3. 以用户为中心：从用户/业务视角描述结果，而非系统内部
4. 可验证：无需了解实现细节即可测试/验证

良好示例：

- “用户可在 3 分钟内完成结账”
- “系统支持 10,000 并发用户”
- “95% 的搜索在 1 秒内返回结果”
- “任务完成率提升 40%”

不佳示例（过于实现导向）：

- “API 响应时间 < 200ms”（过于技术化，改为“用户几乎即时看到结果”）
- “数据库可处理 1000 TPS”（实现细节，改为面向用户的指标）
- “React 组件渲染高效”（框架特定）
- “Redis 缓存命中率 > 80%”（技术特定）
