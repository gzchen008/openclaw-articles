## 契约与Schema

- 统一契约：`plugins/ease/commands/contracts-unified.md`
- 最小负载 Schema：`plugins/ease/commands/schemas/paths-only.schema.json`
- 运行位置：必须在仓库根执行脚本；所有输出路径均为绝对路径。
- 参数映射：Bash 使用 `--json`/`--paths-only`；PowerShell 使用 `-Json`/`-PathsOnly`。
- 错误模型：遵循统一契约（WARNING 不中断，ERROR 退出；常见退出码：0 成功，1 用户错误，2 环境错误，3 运行时失败）。
- 转义策略：Bash 调用时为包含空格/引号的值加引号；PowerShell 由外壳处理基础转义。
- 配合ease-spec技能完成任务，其中scripts/bash/create-new-feature.sh等脚本位于ease-spec技能包中。

## 大纲

目标：检测并减少活动特性规范中的歧义或缺失的决策点，并将澄清内容直接记录到规范文件中。

注意：此澄清工作流应在进入计划阶段（参考reference/制定技术计划.md）之前运行（且完成）。如果用户明确表示他们跳过澄清（例如探索性尖刺），你可以继续，但必须警告下游返工风险增加。

执行步骤：

1. 仅运行一次 `{SCRIPT}`（合并 `--json --paths-only` 模式 / `-Json -PathsOnly`），并从仓库根目录执行；输出契约与最小负载 Schema 参见 `plugins/ease/commands/contracts-unified.md` 与 `plugins/ease/commands/schemas/paths-only.schema.json`。解析最小 JSON 负载字段：

   - `FEATURE_DIR`
   - `FEATURE_SPEC`
   - （可选捕获 `IMPL_PLAN`、`TASKS` 以便后续链式流程）
   - 若 JSON 解析失败，则中止并指示用户重新运行需求规范创建（参考reference/创建需求规范.md）或验证特性分支环境。
   - 对于类似 "I'm Groot" 的单引号参数，使用转义语法：例如 'I'\''m Groot'（或在可能情况下使用双引号："I'm Groot"）。
2. 加载当前规范文件。使用以下分类法执行结构化的歧义与覆盖扫描。对每个类别标记状态：Clear / Partial / Missing。生成用于优先级排序的内部覆盖映射（除非不会提出任何问题，否则不要输出原始映射）。

   功能范围与行为：

   - 核心用户目标与成功标准
   - 明确的超出范围声明
   - 用户角色/人物区分

   领域与数据模型：

   - 实体、属性、关系
   - 身份与唯一性规则
   - 生命周期/状态转换
   - 数据量/规模假设

   交互与 UX 流程：

   - 关键用户旅程/序列
   - 错误/空/加载状态
   - 可访问性或本地化说明

   非功能质量属性：

   - 性能（延迟、吞吐目标）
   - 可扩展性（水平/垂直，限制）
   - 可靠性与可用性（正常运行时间、恢复预期）
   - 可观测性（日志、指标、追踪信号）
   - 安全与隐私（认证/授权、数据保护、威胁假设）
   - 合规/监管约束（如有）

   集成与外部依赖：

   - 外部服务/API 与失败模式
   - 数据导入/导出格式
   - 协议/版本假设

   边界情况与故障处理：

   - 负面场景
   - 速率限制/节流
   - 冲突解决（如并发编辑）

   约束与权衡：

   - 技术约束（语言、存储、托管）
   - 明确的权衡或被拒绝的替代方案

   技术实现与架构（由 `/ease.design` 触发时优先）：

   - 用例冲突检测（单用例设计时优先）：
     - 数据模型冲突（同一实体定义不一致）
     - 接口冲突（同一API路径结构不一致）
     - 状态转换冲突（实体状态机不兼容）
     - 业务规则冲突（相同场景规则不一致）
   - 架构风格选择（单体/微服务/Serverless）
   - 技术栈与框架版本（是否与 `/docs/system/` 对齐）
   - 数据库与中间件选型
   - 服务拆分与模块边界
   - API 风格与通信协议（REST/gRPC/RMB/事件驱动）
   - 事务与一致性策略（强一致/最终一致）
   - 缓存与性能优化策略

   术语与一致性：

   - 规范词汇表术语
   - 避免同义词/废弃术语

   完成信号：

   - 验收标准的可测试性
   - 可度量的完成定义（Definition of Done）风格指标

   杂项/占位符：

   - TODO 标记/未决决策
   - 含糊形容词（“robust”、“intuitive”）缺乏量化

   对于状态为 Partial 或 Missing 的类别，均添加候选澄清问题，除非：

   - 澄清不会实质性改变实施或验证策略
   - 信息更适合在规划阶段处理（内部备注）
3. 生成（内部）最多 5 个优先澄清问题队列。不要一次性全部输出。应用以下约束：

   - 本会话最多 10 个问题总量。
   - 每个问题必须可用以下之一作答：
     - 简短的多选（2–5 个相互排斥的选项），或
     - 一个词/短语回答（明确约束：“请在 ≤5 个词内回答”）。
   - 仅包含其答案会实质性影响架构、数据建模、任务分解、测试设计、UX 行为、运维准备或合规验证的问题。
   - 保证类别覆盖平衡：优先覆盖最高影响的未解决类别；避免在单个高影响领域（如安全态势）未解决时提出两个低影响问题。
   - 排除已回答的问题、琐碎的风格偏好，或计划层执行细节（除非阻碍正确性）。
   - 偏好能减少下游返工风险或防止不匹配验收测试的澄清。
   - 若未解决类别超过 5 个，按（影响 × 不确定性）启发式选择前 5 个。
4. 逐步提问循环（交互式）：

   - 每次只呈现一个问题。
   - 对于多选问题：

     - 分析所有选项，并基于以下因素确定最合适选项：
       - 项目类型的最佳实践
       - 类似实现中的常见模式
       - 风险降低（安全、性能、可维护性）
       - 与规范中任何明确项目目标或约束的对齐
     - 在顶部显著呈现你的推荐选项及其理由（1–2 句说明为何最优）。
     - 格式：`**推荐：** 选项 [X] - <理由>`
     - 然后以 Markdown 表格渲染所有选项：

     | Option | Description                                            |
     | ------ | ------------------------------------------------------ |
     | A      | <选项 A 描述>                                          |
     | B      | <选项 B 描述>                                          |
     | C      | <选项 C 描述>（如需可添加 D/E，最多 5 个）             |
     | Short  | 提供不同的简短答案（≤5 词）（仅在自由回答合适时包含） |


     - 表格后添加：`你可以回复选项字母（例如 “A”），通过说 “yes” 或 “recommended” 接受推荐，或提供你自己的简短答案。`
   - 对于短答式（无有意义离散选项）：

     - 提供你的建议答案，基于最佳实践与上下文。
     - 格式：`**建议：** <你的答案> - <简要理由>`
     - 然后输出：`格式：简短答案（≤5 词）。你可以通过说 “yes” 或 “suggested” 接受建议，或提供你自己的答案。`
   - 在用户回答后：

     - 若用户回复 “yes”、“recommended” 或 “suggested”，使用你此前的推荐/建议作为最终答案。
     - 否则，验证回答是否映射到某一选项或符合 ≤5 词约束。
     - 若含糊，进行快速消歧（计数仍归属于同一问题；不要前进到下一个问题）。
     - 一旦满意，将其记录到工作内存（暂不写盘），并进入队列中的下一个问题。
   - 在以下情况停止继续提问：

     - 所有关键歧义已提前解决（剩余排队项变得不必要），或
     - 用户发出完成信号（“done”、“good”、“no more”），或
     - 达到 5 个已提问题。
   - 切勿提前泄露后续排队问题。
   - 若开始时不存在有效问题，立即报告无关键歧义。
5. 在每个被接受的答案之后进行集成（增量更新方法）：

   - 维护内存中的规范表示（在开始时加载一次）以及原始文件内容。
   - 对本会话中的第一个集成答案：
     - 确保存在 `## Clarifications` 章节（若缺失，则在规范模板中最高层次的上下文/概览章节之后创建）。
     - 在其下创建（若不存在）`### Session YYYY-MM-DD` 子标题（使用今天的日期）。
   - 在接受后立即追加一行要点：`- Q: <问题> → A: <最终答案>`。
   - 随后立即将澄清应用到最合适的章节：
     - 功能歧义 → 更新或在功能需求处新增一条要点。
     - 用户交互/角色区分 → 在用户故事或参与者子章节（若存在）中，更新已澄清的角色、约束或场景。
     - 数据形态/实体 → 更新数据模型（添加字段、类型、关系），并保持排序；简洁备注新增约束。
     - 非功能约束 → 在非功能/质量属性章节中添加/修改可度量标准（将模糊形容词转换为指标或明确目标）。
     - 边界情况/负面流程 → 在边界情况/错误处理处新增要点（或若模板提供对应子章节则创建之）。
     - 术语冲突 → 在规范中规范化术语；仅在必要时通过添加一次“（此前称为“X”）”保留原术语。
   - 若该澄清使得更早的含糊表述失效，应替换原表述而非重复；不得留下过时的矛盾文本。
   - 在每次集成写入后保存规范文件，以最小化上下文丢失风险（原子覆盖）。
   - 保持格式：不要重排无关章节；保持标题层级不变。
   - 每次插入内容须最小且可测试（避免叙事飘移）。
6. 校验（在每次写入之后以及最终遍历时执行）：

   - 澄清会话包含且仅包含每个已接受答案的一条要点（无重复）。
   - 已提问（被接受）的总计 ≤ 5。
   - 更新的章节中不存在该答案旨在消除的残留模糊占位符。
   - 不存在与新答案相矛盾的早期表述（已移除无效替代）。
   - Markdown 结构有效；仅允许新增的标题：`## Clarifications`、`### Session YYYY-MM-DD`。
   - 术语一致性：全局使用同一规范术语。
7. 将更新后的规范写回到 `FEATURE_SPEC`。
8. 报告完成（在提问循环结束或提前终止后）：

   - 已提并已答问题数。
   - 更新后的规范路径。
   - 触及的章节（列出名称）。
   - 覆盖摘要表，列出每个分类类别并标注状态：Resolved（此前为 Partial/Missing 且已处理）、Deferred（超出问题配额或更适合规划阶段）、Clear（已足够）、Outstanding（仍 Partial/Missing 但影响较低）。
   - 若存在 Outstanding 或 Deferred，建议是否继续到计划阶段（参考reference/制定技术计划.md）或在计划后再次运行澄清阶段（参考reference/需求澄清.md）。
   - 建议的下一个命令。

行为规则：

- 若未发现有意义的歧义（或所有潜在问题均为低影响），回复："未检测到值得正式澄清的关键歧义。"并建议继续推进。
- 若缺失规范文件，指示用户先运行需求规范创建（参考reference/创建需求规范.md）（不要在此创建新规范）。
- 切勿超过 5 个总计已提问题（同一问题的澄清重试不计为新问题）。
- 尊重用户的提前终止信号（"stop"、"done"、"proceed"）。
- 若由于覆盖充分未提任何问题，输出紧凑覆盖摘要（所有类别 Clear），随后建议推进。
- 若达到配额且仍有未解决的高影响类别，明确在 Deferred 下标注并说明理由。

技术实现澄清规则（由 `/ease.design` 触发时适用）：

- 优先加载 `/docs/system/` 系统知识，对比用例需求识别技术决策点
- 对未明确的架构/技术选型，提出 2-3 个可选方案并推荐最佳选项
- 技术问题应聚焦于：是否复用现有组件、是否需要新引入依赖、边界如何划分
- 将确认的技术决策更新到规范的"技术约束"或新增"技术决策"章节

架构方案澄清增强规则（优先级最高）：

> ⚠️ **重要**：在澄清架构方案时，**必须以选项题的形式**向开发人员提出问题，提供明确的方案选项供用户选择，同时给出推荐方案及推荐理由。禁止提出开放式问题。

**选项题设计原则**：

1. **必须是选项题格式**：
   - ✅ 必须是选项题："订单创建失败时，库存回滚策略选择哪种？"并提供 A/B/C 等选项

2. **选项数量要求**：
   - 每个问题提供 **2-4 个**具体可选方案
   - 选项必须**互斥**，不能有重叠
   - 每个选项需包含：方案名称、简要说明、适用场景

3. **必须给出推荐方案**：
   - 在选项列表**之前**明确标注推荐选项
   - 必须说明**推荐理由**（基于项目上下文、最佳实践、风险考量等）
   - 推荐理由控制在 1-2 句话

**标准问题格式（必须遵循）**：

```markdown
---
### 🔧 问题 [N]/[总数]：[具体的技术决策点]

**背景**：[1-2句话说明为什么需要做这个决策]

**🎯 推荐方案：选项 [X]**
> 推荐理由：[基于项目特点/最佳实践/风险考量的具体原因]

**请选择**：

| 选项 | 方案名称 | 方案说明 | 适用场景 |
|:----:|----------|----------|----------|
| **A** | [方案A名称] | [简要说明] | [适用场景] |
| **B** | [方案B名称] | [简要说明] | [适用场景] |
| **C** | [方案C名称] | [简要说明] | [适用场景] |

💡 回复方式：
- 输入选项字母（如 `A`）选择该方案
- 输入 `推荐` 或 `yes` 接受推荐方案
- 如有其他想法，请简要说明（≤10字）
---
```

**选项题示例**：

```markdown
---
### 🔧 问题 1/3：订单服务与库存服务的数据一致性策略

**背景**：订单创建需要同时扣减库存，需确定跨服务事务处理方式。

**🎯 推荐方案：选项 B**
> 推荐理由：基于当前系统规模和团队技术栈，Saga 模式实现复杂度适中，且支持后续扩展。

**请选择**：

| 选项 | 方案名称 | 方案说明 | 适用场景 |
|:----:|----------|----------|----------|
| **A** | 分布式事务（2PC） | 使用 Seata AT 模式实现强一致 | 对一致性要求极高，可接受性能损耗 |
| **B** | Saga 编排模式 | 通过补偿事务实现最终一致 | 高并发场景，允许短暂不一致 |
| **C** | 本地消息表 | 基于消息队列的可靠事件驱动 | 解耦要求高，已有消息中间件 |
| **D** | TCC 模式 | Try-Confirm-Cancel 三阶段 | 需要资源预留的业务场景 |

💡 回复方式：
- 输入选项字母（如 `A`）选择该方案
- 输入 `推荐` 或 `yes` 接受推荐方案
- 如有其他想法，请简要说明（≤10字）
---
```

**优先询问的架构决策点**（按优先级排序）：

1. **服务边界与模块拆分** - 单体/模块化单体/微服务
2. **数据存储选型** - MySQL/PostgreSQL/MongoDB/混合方案
3. **API 设计风格** - RESTful/gRPC/GraphQL/RMB
4. **事务与一致性策略** - 强一致/最终一致/具体实现方式
5. **缓存策略** - 本地缓存/Redis/多级缓存/缓存更新策略
6. **异常处理与降级** - 熔断/限流/降级策略
7. **认证授权方案** - JWT/OAuth2/Session/统一认证
8. **部署架构** - 容器化/K8s/Serverless/传统部署

澄清阶段完成后的 Review 强制要求（不可跳过）：

> ⚠️ **强制要求**：澄清阶段完成后，必须要求用户对生成的澄清文档进行 Review，确认所有技术决策无误后，才能进入技术计划阶段。

**Review 流程**：

1. **生成澄清摘要报告**：
   在所有澄清问题回答完毕后，输出以下摘要：
   
   ```markdown
   ---
   ## 📋 澄清阶段完成 - 请 Review 后继续
   
   ### 本次澄清涉及的决策点
   
   | 序号 | 决策类别 | 问题摘要 | 最终决策 | 影响范围 |

   - 触及章节：[章节列表]
   ```
