# NFR 判断指南

## 处理范围

✅ **应处理**（代码/配置层面的技术需求）:
- 性能优化（响应时间、吞吐量、资源占用）
- 代码重构（解耦、可读性、可测试性）
- 依赖升级（框架、库、中间件客户端）
- 安全加固（漏洞修复、加密、鉴权）
- 可观测性（日志、监控埋点、链路追踪）

🚫 **应拒绝**（业务层面需求）:
- 新增业务功能
- 业务流程变更
- UI/交互改动
- 业务规则调整

## 边界判断

```
问自己：这个需求改的是"怎么做"还是"做什么"？
  - "怎么做"（技术实现）→ ✅ 处理
  - "做什么"（业务功能）→ 🚫 拒绝
```

## 拒绝话术

```
该需求涉及业务功能，不在技术需求分析范围内。
如有底层技术支撑需求（如性能、安全），请单独提出。
```
