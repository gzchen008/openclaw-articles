# Flow Execution Standards

本文档为 ease-analysis 所有 flow 提供统一的任务管理、执行标准和生成要求，避免重复内容。

## 任务清单创建

**在开始第一阶段之前，使用 TodoWrite 工具创建完整的任务清单：**

每个任务必须包含：`content`（描述）、`priority`（优先级），初始状态为 `pending`。

示例结构：
```
**阶段1：[阶段名称]（N个核心任务 + 1个质量检查）**
- 任务1描述 - 优先级：高
- 任务2描述 - 优先级：高
- 阶段1质量检查：[检查点描述] - 优先级：高
```

> TodoWrite 详细标准请参见 `task-management-standards.md`

## 任务执行指南

**在整个分析过程中，严格遵循以下执行原则：**

1. **单任务执行**：同一时刻只将一个任务标记为 `in_progress`
2. **及时更新状态**：完成当前任务后立即更新为 `completed`，然后开始下一个任务
3. **质量检查点**：每个阶段的 `-qc` 任务必须在该阶段所有任务完成后才能开始
4. **进度报告**：每完成3-5个任务后，向用户输出当前进度（如：已完成 8/20 任务）
5. **异常处理**：如果某个任务无法完成，标记为 `cancelled` 并在任务描述中记录原因
6. **自动连续执行**：报告进度后**立即继续下一批任务**，无需等待用户确认（详见 SKILL.md）

## 严格生成要求

### 必须遵守的行为

**必须严格遵循以下行为准则：**

1. **基于输入填充内容**：
   - 每个字段、每个章节都必须根据源代码/BRD 分析的真实结果填充
   - 领域名称必须是具体的业务领域（如 "订单管理"、"用户认证"）
   - 实体/用例/规则都必须是具体的概念，不能是示例占位符
   - 所有关系、属性、方法都必须基于实际的业务需求或代码实现

2. **完整填充所有模板章节**：
   - 生成的文档必须包含模板定义的所有章节
   - 每个章节都必须有具体内容，不能留空

3. **确保内容质量和完整性**：
   - 使用具体明确的业务表述
   - 所有描述都应反映实际业务或代码逻辑

### 内容填充检查点原则

**在生成文档时，必须逐项检查以下内容：**

- ✓ 所有必填字段都已填写
- ✓ 所有内容都是基于实际分析结果，非模板示例
- ✓ ID/编号按照规范分配
- ✓ 所有表格都有完整的数据
- ✓ 引用关系清晰可追溯

## 质量改进流程

**当发现需要改进的内容时，必须执行以下流程：**

1. **重新执行相关任务**：
   - 返回需要改进的任务
   - 补充缺失的内容
   - 完善文档质量

2. **完善文档内容**：
   - 确保所有必填字段都有具体内容
   - 确保所有ID按照规范分配
   - 确保所有表格都有完整的数据

3. **更新 todolist 状态**：
   - 将任务状态更新为正确状态
   - 确保进度反映实际情况

4. **向用户说明情况**：
   - 说明需要改进的内容
   - 说明已采取的措施
   - 说明最终结果

## 进度可视化

在阶段完成时输出简洁进度（格式：`📊 进度: X/N | 阶段Y完成 | ⏳ 自动继续...`），**报告后自动进入下一阶段**。仅在全部完成或遇到致命错误时暂停。
