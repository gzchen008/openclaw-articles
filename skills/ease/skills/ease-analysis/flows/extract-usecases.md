### 任务执行指南

**在整个分析过程中，严格遵循以下执行原则：**

1. **单任务执行**：同一时刻只将一个任务标记为 `in_progress`
2. **及时更新状态**：完成当前任务后立即更新为 `completed`，然后开始下一个任务
3. **质量检查点**：每个阶段的 `-qc` 任务必须在该阶段所有任务完成后才能开始
4. **进度报告**：每完成3-5个任务后，向用户输出当前进度（如：已完成 8/20 任务）
5. **异常处理**：如果某个任务无法完成，标记为 `cancelled` 并在任务描述中记录原因
6. **自动连续执行**：报告进度后**立即继续下一批任务**，无需等待用户确认（详见 SKILL.md）

### ✅ 必须遵守的行为（Required Behaviors）

**必须严格遵循以下行为准则：**

1. **基于代码分析结果填充内容**：
    - 每个字段、每个章节都必须根据源代码分析的真实结果填充

    - 用例文档必须包含模板定义的所有章节：头部元数据、核心信息、场景流程、其他要求
    - 规则文档必须包含模板定义的所有章节：头部元数据、规则目录、详细规则说明、约束条件、成功标准

    - 每个用例的主流程必须包含至少 3 个具体的操作步骤

4. **保持用例与规则的严格分离**：
    - 用例文档（usecase.md）中只能引用规则ID
    - 规则文档（rules.md）中必须包含所有规则的完整详细信息

## 处理工作流（Processing Workflow）

> ⚠️ **重要提示：** 本流程的输出必须符合 `analyze-brd.md` 的要求：
> - 输出到 `docs/domains/[编号]-[module_name]/usecases/[编号]-[subdomain]/` 目录
> - 每个子领域目录下固定包含两个文件：`usecase.md` 和 `rules.md`
> - 子领域目录必须带三位编号（如 `001-authentication`）

### 阶段 1：准备与领域发现（PREPARATION & DOMAIN DISCOVERY）

#### 1.1 初始分析（Initial Analysis）
- **代码库理解（Codebase Understanding）：**
    - 通读整个代码库以理解核心业务目标
    - 识别系统架构与主要组件
    - 记录系统为谁解决了什么问题

- **领域模型发现（Domain Model Discovery）：**
    - 首选来源：阅读 `docs/domains/[编号]-[module_name]/` 下的领域模型文档
    - 备选方法：从代码结构、配置文件、包组织中推导领域
    - 构建领域术语表（统一语言，Ubiquitous Language）
    - 识别子领域（subdomain）边界，为后续用例组织做准备

#### 1.2 现有文档审阅（Existing Documentation Review）
- 检查 `docs/domains/[编号]-[module_name]/usecases/` 中的既有用例文档
- 审阅代码注释与注解以获取业务语境
- 记录已有的业务规则文档

#### 1.3 参与者识别（Actor Identification）
- **主参与者（Primary Actors）：** 发起动作的角色（如 Customer、Manager）
- **次参与者（Secondary Actors）：** 服务提供方或信息接收方（如 Payment Gateway、ERP）
- 记录每个参与者的角色与系统关系

### 阶段 2：用例识别（USE CASE IDENTIFICATION）

#### 2.1 参与者目标发现（Actor Goals Discovery）
- 针对每个主参与者询问："该参与者希望达成什么？"
- 以"动词 + 宾语"的格式表达目标
- 示例： "Browse Products"、"Place Order"、"Review Refunds"

#### 2.2 用例定义（Use Case Definition）
- 将参与者目标转化为用例
- 分配标识符：`UC-[%03d]`（例如 UC-001）
- 创建简明、以行动为导向的名称
- **按子领域（subdomain）组织用例**
- **扫描现有子领域目录，获取下一个子领域编号（三位数字）**

#### 2.3 用例文档（Use Case Documentation）
针对每个用例，记录：

**a) 基础流程（Basic Flow / Happy Path）：**
- 至少包含 3 个操作步骤
- 清晰、逐步描述成功路径
- **在需要应用业务规则的地方，明确标注规则 ID（如 `Use BR-001`），不复制规则细节**

**b) 备选流程（Alternative Flows）：**
- 能够实现目标的有效变体
- 记录分支条件
- **在需要应用业务规则的地方，明确标注规则 ID**

**c) 异常流程（Exception Flows）：**
- 错误条件与失败场景
- 覆盖代码中识别的所有边界情况
- **在需要应用业务规则的地方，明确标注规则 ID**

**d) 前置条件（Pre-conditions）：**
- 执行前所需的系统状态
- 参与者的认证/授权要求

**e) 后置条件（Post-conditions）：**
- 成功完成后的系统状态
- 被触发的领域事件

**f) 依赖关系（Dependencies）：**
- 清晰描述与其他用例的关系
- 抽象共用流程以便复用

**g) 业务规则引用（Business Rules References）：**
- 列出在流程中引用的所有业务规则 ID
- **重要：** 仅列出规则 ID，规则详细信息在 rules.md 中维护

### 阶段 3：业务规则抽取（BUSINESS RULE EXTRACTION）

#### 3.1 规则挖掘（Rule Mining）
- 审阅用例步骤中的嵌入逻辑
- 识别校验、计算与触发条件
- 从源代码分析中抽取

#### 3.2 规则分类（Rule Classification）
- **Constraints（约束）：** 阻止行为的限制
- **Computations（计算）：** 计算值的公式
- **Inferences（推断）：** If-Then 形式的逻辑规则
- **Action Enablers（动作触发）：** 触发后续动作的规则

#### 3.3 业务规则目录（Business Rules Catalog）
- **格式：** `BR-[%03d]-[rule-name]`（1-3 个词的名称）
- **示例：** `BR-001-user-auth`、`BR-002-order-modify`
- **内容要求：**
    - 完整的规则描述（避免含糊表述）
    - 类型分类（Constraints、Computations、Inferences、Action Enablers）
    - 关联用例引用
    - **关键要求：**
      - 在用例文档（usecase.md）中仅按 ID 引用规则，绝不复制规则细节
      - 所有规则详细信息必须写入 rules.md 文件
      - 保持用例和规则的严格分离

3. **规则描述：** 
   - 完整的规则描述，避免含糊表述
   - 明确规则类型、触发条件、执行逻辑
   - 包含关联用例引用
4. **约束条件：** 明确列出业务约束和技术约束
5. **成功标准：** 定义可量化的成功标准

#### 分离原则（Separation Principle）

- **严格分离：** 用例文档（usecase.md）和规则文档（rules.md）必须严格分离
- **引用而非复制：** 用例文档中只引用规则 ID，不复制规则内容
- **单一来源：** 规则详细信息只在 rules.md 中维护，确保单一数据源

## 执行指南（Execution Guidelines）

### 源代码范围（Source Code Scope）
**包含（Include）：**
- 业务逻辑实现
- API 端点/控制器
- 服务层组件
- 领域模型与实体
- 影响业务规则的配置

**排除（Exclude）：**
- 测试文件（*_test.*、*.test.*、/test/、/tests/）
- 构建产物（/build/、/dist/、/target/）
- 前端/UI 逻辑
- 临时文件

### 关键原则（Key Principles）
1. **关注点分离（Separation of Concerns）：** 
   - 保持业务规则与用例流程严格独立
   - 用例文档（usecase.md）和规则文档（rules.md）必须分离
   - 用例中仅引用规则 ID，不复制规则内容
2. **可追溯性（Traceability）：** 每个元素必须可追溯到源代码
3. **领域语言（Domain Language）：** 一致使用行业特定术语
4. **完整性（Completeness）：** 覆盖所有参与者、流程与边界情况
5. **可维护性（Maintainability）：** 随代码演进易于更新的结构
6. **统一标准（Unified Standards）：** 
   - 所有用例文档遵循 `usecases-template.md` 模板
   - 所有规则文档遵循 `rules-template.md` 模板
   - 确保输出格式和结构的一致性

## 错误处理（Error Handling）
- 无法确定领域：创建 `uncategorized` 领域
- 信息冲突：在用例备注部分记录
- 依赖循环：标记以供人工审阅
- 缺失业务语境：标注为 “requires clarification”

## 成功标准（Success Criteria）
当满足以下条件时，抽取工作完成：
1. 所有生产代码均已分析
2. 每个业务操作都有对应用例
3. 全部用例通过质量检查
4. 依赖关系清晰映射
5. 业务规则得到正确引用且不重复
6. 文档一致遵循模板结构
7. **用例和规则已正确分离到 usecase.md 和 rules.md 文件**
8. **所有子领域目录结构符合 `analyze-brd.md` 的要求**
9. **用例文档中仅引用规则 ID，规则详细信息在 rules.md 中**
