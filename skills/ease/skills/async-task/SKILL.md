# 云端异步任务提交技能

## Overview

将用户任务提交到云端后台异步执行的技能。系统通过 GitHub Issue 机制实现云端异步任务队列，此底层实现细节对用户完全透明。用户感知的是"任务已提交到云端后台执行"。

## When to Use

### 显式触发（推荐）

用户使用以下**显式标记**时，直接触发此技能：

| 触发方式 | 示例 |
|----------|------|
| `--async` 标记 | `--async 分析整个项目代码` |
| `[@ease]` 标记 | `[@ease] 生成性能优化报告` |
| `提交后台:` 前缀 | `提交后台：批量数据迁移` |
| `异步执行:` 前缀 | `异步执行：全量代码审查` |
| `云端处理:` 前缀 | `云端处理：运行回归测试` |

### 意图识别触发

当用户请求**同时满足**以下条件时，触发此技能：

1. **包含异步相关关键词**：
   - 后台任务、异步执行、云端处理、稍后处理、排队、异步、后台
   - 长时间运行、耗时操作、批量处理、离线处理

2. **包含任务意图动词**：
   - 分析、生成、处理、执行、运行
   - 审查、重构、优化、迁移、扫描

3. **排除误命中场景**（见下方防误命中规则）

### 防误命中规则

**以下场景不触发此技能**：

| 场景 | 说明 | 正确处理方式 |
|------|------|--------------|
| 普通代码问题 | "异步函数报错了" | 使用 ease:systematic-debugging |
| 日常开发任务 | "写个异步方法" | 直接实现，无需提交后台 |
| 简单查询 | "什么是异步" | 使用 ease:helper 知识问答 |
| Git 操作相关 | "git 提交问题" | 使用 ease:git 命令 |
| Issue 讨论相关 | "查看 issue 状态" | 引导用户直接访问 GitHub |

**判断技巧**：如果用户只是想**了解**或**讨论**异步相关概念，而非**提交任务到云端执行**，则不触发。

### 典型应用场景

**适合提交到云端后台的任务**：
- 需要长时间运行的分析任务（全项目代码分析、依赖分析）
- 大批量数据处理（数据迁移、批量转换）
- 完整的测试套件执行（回归测试、集成测试）
- 深度代码审查与优化（性能分析、安全扫描）
- 大规模文档生成（API 文档、架构文档）
- 复杂的架构重构（模块拆分、技术栈迁移）

## 用户视角 vs 技术实现

### 用户视角（对外）
- 任务已提交到云端后台
- 任务正在排队处理
- 任务追踪 ID
- 任务状态：已排队 / 处理中 / 已完成

### 技术实现（内部）
- 通过 GitHub Issue API 创建任务
- Issue 类型映射：需求(6) / 任务(5) / 缺陷(7)
- 使用标准化的标题和描述格式
- 获取项目名和系统用户名

## 接口信息

### 请求接口

| 项目         | 说明                                              |
| ------------ | ------------------------------------------------- |

| Method       | `POST`                                          |
| Content-Type | `application/json`                              |

| 参数名       | 类型   | 必填 | 说明                                                            |

| 用户输入类型        | issue_type | 标题前缀   | 说明                         |
| ------------------- | ---------- | ---------- | ---------------------------- |
| 新功能请求/改进建议 | 6          | `[feature]` | Feature、Improvement         |
| 技术任务/重构/优化  | 5          | `[task]`    | Refactor、Performance、Chore |

## 实现步骤

### Step 1: 获取系统用户名

根据操作系统使用对应命令获取当前系统用户名：

**macOS/Linux:**

```bash
whoami
```

**Windows PowerShell:**

```powershell
$env:USERNAME
```

**通用方式（跨平台）:**

```bash
echo $USER  # macOS/Linux
echo %USERNAME%  # Windows CMD
```

### Step 2: 获取项目名

从本地 git 仓库信息获取项目名：

**macOS/Linux:**

```bash
# 方法1: 从远程仓库 URL 获取项目名
git remote get-url origin | xargs basename -s .git

# 方法2: 如果远程 URL 不可用，从当前目录获取
basename $(git rev-parse --show-toplevel)
```

**Windows PowerShell:**

```powershell
# 方法1: 从远程仓库 URL 获取项目名
$remoteUrl = git remote get-url origin
$projectName = ($remoteUrl -split '/')[-1] -replace '\.git$', ''

# 方法2: 如果远程 URL 不可用，从当前目录获取
$projectName = Split-Path -Leaf (git rev-parse --show-toplevel)
```

**Windows CMD:**

```cmd
REM 从远程仓库 URL 获取项目名
FOR /f "delims=" %i IN ('git remote get-url origin ^| xargs basename -s .git') DO SET PROJECT_NAME=%i
```

### Step 2.5: 获取当前分支名

> **⚠️ 重要**：分支信息是云端任务追踪和复现的关键字段，帮助任务执行者定位到正确的代码库版本。

从本地 git 仓库获取当前分支名：

**macOS/Linux:**

```bash
# 方法1: 使用 git branch --show-current（推荐，Git 2.22+）
git branch --show-current

# 方法2: 使用 git rev-parse --abbrev-ref HEAD
git rev-parse --abbrev-ref HEAD
```

**Windows PowerShell:**

```powershell
# 方法1: 使用 git branch --show-current（推荐，Git 2.22+）
git branch --show-current

# 方法2: 使用 git rev-parse --abbrev-ref HEAD
git rev-parse --abbrev-ref HEAD
```

**Windows CMD:**

```cmd
REM 获取当前分支名
FOR /f "delims=" %i IN ('git branch --show-current') DO SET BRANCH_NAME=%i
```

### Step 2.5.1: 判定目标分支与远程分支就绪要求

> **⚠️ 强制前置条件**：云端任务不只依赖任务描述，还依赖目标分支在 remote 可访问。默认目标分支是当前分支；如果用户明确指定了其他分支，则以用户指定分支为准。
>
> 在正式提交云端任务前，必须检查该目标分支是否已经存在于远程仓库：
> - **如果远程分支不存在，则不得提交云端任务**
> - 必须明确提醒用户先 push 代码，使远程分支就绪后再重试
> - 这不是普通建议，而是提交前必须满足的门禁条件
>
> 需要同时区分两层语义：
> 1. **远程分支存在性**：硬门禁；不存在就不能提交云端任务
> 2. **本地与远程是否一致**：强提醒；如果本地还有未提交修改，云端看到的是远程版本，不是当前本地改动

**检查示例：**
```bash
TARGET_BRANCH=$(git branch --show-current)
git ls-remote --heads origin "$TARGET_BRANCH"
```

### Step 3: 确定任务类型

根据用户输入的内容判断任务类型并设置对应的 `issue_type`：

| 用户输入特征             | issue_type | 标题前缀   | 示例关键词                   |
| ------------------------ | ---------- | ---------- | ---------------------------- |
| 新功能请求、新增XX功能   | 6          | `[feature]` | 添加、新增、实现、支持       |
| 改进建议、优化建议、增强 | 6          | `[feature]` | 优化、改进、增强、建议       |
| Bug 报告、修复、错误     | 7          | `[bug]`     | 修复、错误、异常、失败、问题 |
| 重构、性能优化、技术债   | 5          | `[task]`    | 重构、性能、代码质量         |
| 配置变更、环境调整       | 5          | `[task]`    | 配置、部署、环境             |
| 其他不明确内容           | 6          | `[feature]` | -                            |

**判断逻辑**：

1. 优先识别 Bug/缺陷类（关键词：修复、错误、异常、失败、问题）
2. 其次识别技术任务类（关键词：重构、性能、配置、部署）
3. 默认归类为需求类（新功能、改进建议）

### Step 3.2: 生成唯一 6 位识别码

在生成标题前，必须先为本次异步任务生成一个可追踪的唯一识别码。

**规则**：
- 长度固定 6 位
- 字符集仅允许 `0-9` 和 `A-Z`
- 示例：`ABC123`
- 必须基于 `.eases/ease_tasks/` 目录中的已有任务记录做本地去重
- 若撞码则重新生成，并限制重试次数避免无限循环

**目录规则**：
- 本地索引目录固定为 `.eases/ease_tasks/`
- 仅在云端提交成功后，才写入单任务 JSON 记录
- 文件命名建议：`<YYYYMMDDHHMMSS>-<IDENTIFIER>.json`

### Step 3.5: 质量检查与智能澄清

> **跳过条件**：当任务由其他命令转发而来时（即 ORIGINAL_COMMAND 以 `/ease:debug`、`/ease:arch-docs`、`/ease:analyze-code`、`/ease:architecture-review`、`/ease:write-docs` 等非 `/ease:async-task` 命令开头），**直接跳过 Step 3.5**，进入 Step 4。这些命令本身已有明确的任务上下文，无需额外质量检查和澄清。

**仅当用户直接使用 `/ease:async-task` 命令时**，才执行以下质量检查与澄清流程。

在提交云端前，对用户输入进行质量检查，必要时进行澄清。此步骤确保任务描述清晰、完整，提高云端执行成功率。

#### 3.5.1 质量检查清单

**基础完整性检查**（必须全部通过才能跳过澄清）：

| 检查项 | 检查方法 | 不满足时的澄清方向 |
|--------|----------|-------------------|
| 任务目标 | 是否明确要做什么 | 询问核心目标 |
| 输入信息 | 是否提供足够上下文 | 询问相关文件/路径/错误信息 |
| 期望输出 | 是否说明预期结果 | 询问期望交付物 |
| 范围边界 | 是否明确任务边界 | 询问包含/排除内容 |

**任务类型特异性检查**（根据 issue_type 进行）：

**功能开发任务（issue_type=6, [feature]）**：
- [ ] 功能的具体使用场景
- [ ] 交互方式（API/UI/命令行）
- [ ] 数据输入/输出格式
- [ ] 验收标准

**Bug 修复任务（issue_type=7, [bug]）**：
- [ ] 复现步骤
- [ ] 预期行为 vs 实际行为
- [ ] 错误日志/堆栈信息
- [ ] 环境信息（版本、配置）

**技术任务（issue_type=5, [task]）**：
- [ ] 具体的技术目标
- [ ] 当前痛点/问题
- [ ] 期望的改进指标
- [ ] 约束条件

**架构一致性检查**（可选）：

当检测到以下情况时，进行架构一致性检查：
- 涉及多个模块/服务的任务
- 可能影响数据模型的变更
- 涉及 API 设计/修改的任务
- 需要引入新依赖/技术的任务

**检查方法**：
```bash
# 读取项目技术架构文档（如果存在）
if [ -f "docs/system/overview.md" ]; then
  # 了解系统架构
fi
if [ -f "memory/constitution.md" ]; then
  # 了解项目约束和技术栈
fi
```

#### 3.5.2 云端可读性风险检查

在正式提交前，必须补充检查“云端任务能否读到、读对用户输入”的风险。此检查用于前置提醒，不是硬性拦截。

**高风险场景识别**（命中任一项都应纳入澄清判断）：
- 用户环境明显是 **Windows PowerShell**
- 用户输入包含**中文内容**，且可能通过 PowerShell 参数或本地文本文件传入
- 用户提到**文件或目录路径**，尤其是绝对路径、桌面/下载目录、`/tmp`、Windows 盘符路径、UNC 路径等
- 路径位于**当前 git 仓库根目录之外**
- 用户只是“提到路径/目录”，但没有把真正需要的内容粘贴进任务描述

**统一告知口径**：
1. **PowerShell 中文乱码风险**
   - Windows PowerShell 下，中文需求、日志或文件内容可能因为终端/文件编码不一致而出现乱码
   - 不能仅因为用户本地文件可读，就默认云端拿到的中文内容也正确无误
   - 如果任务依赖中文原文，优先建议用户直接粘贴关键中文内容；或确认文件采用 UTF-8，并把关键内容并入最终任务描述
   - 可以复用 `Get-Content ... -Encoding UTF8` 的做法，但应将其视为澄清阶段风险提醒，而非普通命令细节

2. **仓库外文件/目录不可直接访问风险**
   - 用户提到的本地文件或目录，尤其是仓库外路径，只存在于用户本机
   - 云端异步执行环境通常无法直接读取这些本地文件或目录
   - 仅仅在任务中提到“去看某个目录/某个文件”，不等于云端真的能访问它
   - 更可靠的做法是：把关键内容直接写入任务描述，或把所需文件纳入仓库后使用仓库相对路径重新提交

#### 3.5.3 澄清触发条件

满足以下**任一**条件时触发澄清：
- **基础完整性检查**：有 2+ 项不满足
- **任务类型特异性检查**：有 1+ 项关键信息缺失
- **云端可读性风险检查**：命中 PowerShell 中文乱码风险，或命中仓库外文件/目录不可直接访问风险
- **描述模糊度**：包含过多模糊词（"一些"、"相关"、"等"）
- **极简输入**：输入长度 < 20 个字符

#### 3.5.4 澄清问题设计

**约束规则**：
- **澄清次数上限**：最多 3 个问题
- **问题格式**：必须为选项题（A/B/C），禁止开放式问题
- **推荐方案**：必须给出推荐选项及理由
- **风险提示表达**：必须使用“建议 + 原因 + 可选行动”，不能做硬性拦截
- **轮次约束**：云端可读性风险提醒计入现有澄清轮次限制，不新增无限追问

**云端可读性风险模板**：

```markdown
### 🔧 问题 [N]/3：云端任务输入可读性确认

**背景**：当前任务描述里存在云端可能读不到或读不对的输入风险，可能表现为中文乱码，或引用了云端无法直接访问的本地文件/目录。

**🎯 推荐方案：选项 B**
> 推荐理由：先把关键内容补充到任务描述里，比让云端猜测本地路径或承受编码风险更稳妥，能明显降低后台处理偏航概率。

| 选项 | 方案名称 | 方案说明 | 适用场景 |
|:----:|----------|----------|----------|
| **A** | 继续提交 | 继续提交，但仅基于当前已提供的文本信息 | 当前描述已足够完整，接受云端按现有文本处理 |
| **B** | 补充内容后提交 | 先补充关键文件内容、中文原文、日志片段或目录摘要，再提交 | 任务依赖关键上下文，当前信息不足以稳定执行 |
| **C** | 改为仓库相对路径 | 先把需要的文件纳入仓库，再使用仓库相对路径重新提交 | 当前依赖仓库外文件或目录，希望云端稳定可访问 |

💡 回复方式：输入 `A`/`B`/`C`，或输入 `推荐` 接受推荐方案
```

**标准问题格式**：

```markdown
---
### 🔧 问题 [N]/3：[具体的问题标题]

**背景**：[1-2句话说明为什么需要澄清这个点]

**🎯 推荐方案：选项 [X]**
> 推荐理由：[基于任务特点/最佳实践/风险考量的具体原因]

**请选择**：

| 选项 | 方案名称 | 方案说明 | 适用场景 |
|:----:|----------|----------|----------|
| **A** | [方案A名称] | [简要说明] | [适用场景] |
| **B** | [方案B名称] | [简要说明] | [适用场景] |
| **C** | [方案C名称] | [简要说明] | [适用场景] |

💡 回复方式：
- 输入选项字母（如 `A`）选择该方案
- 输入 `推荐` 或 `yes` 接受推荐方案

#### 3.5.5 多轮澄清流程

**逐步提问循环**：

```
初始化：clarification_queue = [] (优先级排序的澄清问题)
         max_questions = 3
         asked_count = 0

循环（while asked_count < max_questions and clarification_queue）：
  1. 呈现一个问题（pop 第一个）
  2. 等待用户回答
  3. 验证并记录答案（A/B/C/推荐/done）
  4. 更新任务描述
  5. asked_count += 1

停止条件：
- 已问满 3 个问题
- 所有关键歧义已解决
- 用户发出完成信号（"done"、"good"、"no more"）
```

#### 3.5.7 质量检查通过条件

**直接提交（跳过澄清）**需满足以下所有条件：
1. 基础完整性检查：0-1 项不满足
2. 任务类型特异性检查：0 项关键信息缺失
3. 描述长度 ≥ 50 个字符
4. 不包含大量模糊词（占比 < 20%）

### Step 4: 生成标题和描述

根据用户输入生成标准格式的标题和详细描述：

**标题格式**：`[<type>] <ABC123> <简述>` (使用英文标签: `[feature]`/`[task]`/`[bug]`，识别码为固定 6 位大写字母/数字)

**描述格式**：

```markdown
## 问题描述
<用户输入的详细内容>

## 环境信息
- 项目：<项目名>
- 操作系统：<系统信息>

- 提交时间：<当前时间>
- 提交人：<系统用户名>
- 原始命令：<完整命令>
```

**获取当前时间（跨平台）**：

- macOS/Linux: `date '+%Y-%m-%d %H:%M:%S'`
- Windows PowerShell: `Get-Date -Format 'yyyy-MM-dd HH:mm:ss'`

### Step 5: 提交前仓库就绪检查与请求发送

#### Step 5.1: 远程分支就绪检查（硬门禁）

> **⚠️ 必经步骤**：在发送请求前，必须先确认目标分支已经存在于远程仓库。
>
> - 默认目标分支是当前分支
> - 如果用户明确指定其他分支，则以指定分支为准
> - **目标分支必须已存在于远程仓库**
> - **远程分支不存在时不得提交云端任务**
> - 此时必须明确提示用户：**需先 push 代码**，待远程分支就绪后再重试

```bash
TARGET_BRANCH=$(git branch --show-current)
git ls-remote --heads origin "$TARGET_BRANCH"
```

**阻断式处理要求：**
- 如果未查到远程分支，立即暂停提交流程
- 明确告知“远程分支不存在，因此不得提交云端任务”
- 明确告知“请先 push 代码”
- 如用户希望先整理本地提交，可在用户确认后交接 `/ease:git-commit`
- 但不要承诺 `/ease:git-commit` 会自动 push；push 仍需用户显式确认

#### Step 5.2: 本地与远程一致性检查（强提醒）

> **⚠️ 强提醒**：即使目标分支已存在 remote，如果本地仍有未提交修改，云端处理的也是远程版本，而不是当前本地未提交内容。

```bash
git status --porcelain
```

如果存在未提交修改，应明确提醒：
- 远程分支已就绪，但本地代码与远程代码并不一致
- 若现在继续提交云端任务，云端看到的是远程版本，不是当前本地改动
- 在用户确认后，可以先交接 `/ease:git-commit`，帮助整理本地提交
- `/ease:git-commit` 的行为应以安全语义为准：默认优先处理 staged changes、提交前确认 commit message、push 需显式确认

#### Step 5.3: 请求发送

使用 curl 发送 POST 请求：

```bash
curl -X POST http://omnisight.ease.webank.com/api/issues/create \
  -H "Content-Type: application/json" \
  -d '{
    "project_name": "<项目名>",
    "title": "[feature] <ABC123> <简述>",
    "issue_type": 6,
    "desc": "<详细描述>",
    "user": "<系统用户名>",
    "branch": "<当前分支名>",
    "original_command": "<完整的原始命令字符串>"
  }'

**前置检查（可选）**：
```bash
# 检查当前分支是否有未提交的代码
if [[ $(git status --porcelain) == *"M" ]]; then
  echo "⚠️ 检测到未提交的代码更改"
  echo "当前分支: $(git branch --show-current)"
  echo "建议: 请先使用 /ease:git-commit 提交代码，以确保云端任务执行正确的代码库版本"
  read -p "是否继续提交？ (y/n)"
  if [[ $REPLY != "y" ]]; then
    echo "任务已取消"
    exit 1
  fi
fi
```**注意**：`issue_type` 必须根据 Step 3 判断的类型设置：

- 需求类：`"issue_type": 6`, 标题使用 `[feature]`
- 任务类：`"issue_type": 5`, 标题使用 `[task]`
- 缺陷类：`"issue_type": 7`, 标题使用 `[bug]`

### Step 6: 处理响应并向用户反馈

**检查请求状态**：
1. 检查 HTTP 状态码是否为 2xx
2. 解析服务端返回的响应
3. 仅在服务端返回成功后写入本地任务索引 `.eases/ease_tasks/`
4. 按照用户视角格式化反馈

**本地任务索引字段**：
- `identifier`
- `title`
- `task_content`
- `submission_time`
- `issue_type`
- `username`
- `project_name`
- `branch`
- `original_command`
- 可选：`issue_id`、`url`

**用户反馈格式（成功）**：
```

任务识别码: <ABC123>

```

**用户反馈格式（失败）**：
```

- 网络连接问题
- 云端服务暂时不可用

**重要**：向用户反馈时使用"云端任务"、"后台处理"等术语，**避免**直接提及"Issue"、"GitHub"、"PR"等底层实现细节。

## 执行流程

1. 获取系统用户名
2. 从 git 仓库远程 URL 获取项目名
3. 获取当前 git 分支名
4. 分析用户输入，确定任务类型
5. **质量检查与智能澄清**（仅限 `/ease:async-task` 直接调用）
   - 若由其他命令转发（如 debug、arch-docs 等），**跳过此步骤**
   - 基础完整性检查
   - 任务类型特异性检查
   - 架构一致性检查（可选）
   - 必要时进行澄清（最多 3 个问题）
   - 澄清后等待用户确认
6. 生成标准格式的标题和详细描述
7. **远程分支就绪检查**（目标分支必须已存在于远程仓库）
8. **本地与远程一致性提醒**（如有未提交修改，明确说明云端看到的是远程版本）
9. 构建并发送 POST 请求到接口
10. 处理响应并以用户视角格式化反馈

## 注意事项

### 用户视角（对外）
- **透明化底层实现**：用户感知的是"任务已提交到云端后台"，不需要了解 GitHub Issue 的存在
- **统一反馈术语**：使用"云端任务"、"后台处理"、"异步执行"、"任务队列"等术语
- **避免技术细节**：不要向用户提及"Issue"、"PR"、"GitHub"等底层实现概念

### 技术实现（内部使用）
1. **issue_type 必须正确设置**：根据用户输入内容判断类型，设置为 5（任务）/6（需求）/7（缺陷）
2. **标题格式必须包含类型标签和识别码**：`[feature] <ABC123> 简述` / `[task] <ABC123> 简述` / `[bug] <ABC123> 简述`
3. **识别码生成规则**：固定 6 位，仅允许数字和大写字母，且要基于 `.eases/ease_tasks/` 做本地去重
4. **类型判断优先级**：缺陷类（Bug/修复） > 任务类（重构/性能） > 需求类（新功能/改进）
4. 用户名需要从系统获取（纯用户名，如 "tctp"），不可硬编码
5. 项目名从 git 仓库的远程 URL 中获取（移除 `.git` 后缀）
6. 如果 git 远程仓库不可用，可降级使用 git 仓库根目录名
7. 描述内容需要根据用户输入进行完善和结构化
8. 确保请求中的 special 字符（如引号、换行）正确转义
9. 敏感信息不应包含在任务内容中
10. 如遇网络问题导致请求失败，应向用户说明并建议稍后重试
