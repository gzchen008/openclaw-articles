# Root Cause Tracing — 从失败点向源头回溯

本参考文档提供一套“剥洋葱式”的向后追踪方法，用最小阅读量定位到“错误输入/错误状态”第一次出现的位置。

> ⚠️ **铁律：不要在症状点修复**。必须找到“第一次变坏”的位置再修复。

## 适用场景

- 异常发生在调用栈深处，你只看到了最后一层报错
- 传入的值是错的，但你不知道它来自哪里
- 状态/配置在某个环节被覆盖、丢失或未传播
- 多组件链路（CI -> 构建 -> 脚本 -> 运行时，或 API -> 服务 -> DB）定位困难

## 核心流程

### 1) 锁定失败点（Failing Point）

你必须写下“失败点三要素”：

- **位置**：文件路径 + 行号 + 函数名
- **输入**：关键参数/环境变量/配置项（只列影响失败的）
- **假设**：该函数对输入/状态的假设是什么（例如：非空、格式、范围、已初始化）

输出模板：

```text
失败点：path/to/file.ts:123 in doThing()
失败表现：TypeError: cannot read property 'x' of undefined
关键输入：userId=..., payload=..., ENV.FOO=...
隐含假设：payload.user 必须存在
```

### 2) 沿调用栈向上一层（One Frame Up）

对调用栈的上一层重复“失败点三要素”，直到你能回答：

- 这个错误值/错误状态 **在哪里第一次出现**？

规则：

- 每次只向上一层
- 每一层只阅读“该层函数本体 + 与失败分支直接相关的少量分支”
- 任何时候都不要先通读整个模块或整个目录

### 3) 把“值”当作证据（Value-as-Evidence）

当你看到一个“坏值”（例如空、格式不对、越界）：

1. 写下坏值是什么
2. 写下它应当是什么
3. 在它的生产/传递点加日志或断点（只加最小量）
4. 再跑一次复现，找到“从好变坏”的那一跳

### 4) 明确边界层（Boundary Tracing）

如果跨越组件/边界（HTTP、队列、文件、脚本、环境变量、配置中心）：

对每个边界都记录：

- 输入是什么（序列化前）
- 输出是什么（序列化后）
- 关键配置是否存在、值是否一致
- 权限/身份信息是否传递

目标：定位“哪一个边界把数据/状态弄坏了”。

## 常见模式与定位提示

| 模式 | 典型信号 | 追踪策略 |
|------|----------|----------|
| 期望非空但为 null/undefined | NPE/TypeError | 回溯到第一次赋值/构造处 |
| 配置丢失 | ENV/配置读取失败 | 在读取点与注入点之间打点 |
| 序列化/反序列化错误 | 字段缺失、类型变字符串 | 对比请求体/DTO/Schema |
| 时序问题 | 偶现、重试后成功 | 引入条件等待/增加可观测性 |
| 并发问题 | 竞态、数据互相污染 | 收敛共享状态、加隔离与锁 |

## 完成标准

你完成 root-cause-tracing 的标志不是“我觉得找到了”，而是：

- ✅ 你能指出“第一次变坏”的具体位置（函数/边界）
- ✅ 你能解释为什么它会变坏（与证据对应）
- ✅ 你能写出一个最小实验来再次证明这个因果关系
