# 系统性调试 — 证据驱动的根因定位

## 异步任务检测

> **优先检测**：在执行本技能前，首先检测用户输入是否包含 `[@ease]` 和 `async` 关键词。

**检测条件**：当用户输入**同时满足**以下条件时，触发 `async-task` 技能：
- 包含 `[@ease]` 标记（或 `[@Ease]`、`[@EASE]`）
- 包含 `async` 关键词（或 `--async` 标记）

**触发示例**：
- `[@ease] async 调试登录接口超时问题`
- `/ease:debug [@ease]--async 分析测试失败原因`
- `[@ease] 异步执行：定位根因`

**检测逻辑**：
```bash
# 检测用户输入中是否同时包含 @ease 和 async 关键词
USER_INPUT="$1"  # 用户完整输入
if [[ "$USER_INPUT" =~ @ease|@Ease|@EASE ]] && [[ "$USER_INPUT" =~ async|\[async\]|异步 ]]; then
    # 触发 async-task 技能，将任务提交到云端
    # 任务类型：缺陷类 (issue_type=7, 标题前缀=[bug])
    # 任务内容：完整的用户输入（包括命令）
fi
```

**触发后的行为**：
1. 将完整的用户输入（如 `/ease:debug [@ease]--async 调试问题`）作为任务描述
2. 使用 `async-task` 技能提交到云端后台
3. 向用户反馈"任务已提交到云端后台"

随机修复浪费时间，还会引入新 bug。快速补丁只是在掩盖根本问题。

> **定位**：当你遇到任何 bug、测试失败、构建失败、意外行为或性能问题时，本技能负责把“症状”收敛为“可验证的根本原因”，并给出“最小、可回归验证”的修复路径。

## 🔒 执行契约（Execution Contract）

> ⚠️ **没有根本原因调查就没有修复**，此规则不可协商、不可省略、不可修改。

1. **证据优先**：所有结论必须能指向证据（日志/堆栈/配置差异/可复现步骤/最小用例）。
2. **可重复**：在提出修复之前，必须能“稳定复现”或明确说明为什么无法复现以及已收集的证据。
3. **一次一个变量**：每次实验只改变一个变量，避免堆叠改动。
4. **最小修复**：修复只覆盖根因，不做“既然我在这里”的重构。
5. **验证后再宣称完成**：声称修复成功之前，必须运行可验证的命令（测试/构建/重现步骤）。
6. **3 次失败即停**：同类问题尝试 3 个修复仍失败，必须回到 Phase 1 并质疑架构/假设；禁止尝试第 4 个“碰碰运气”的修复。

## 适用范围

### 适用场景
- 测试失败（单测/集成测试/回归）
- 生产 bug / 线上告警
- 构建失败 / CI 失败
- 意外行为（数据不一致、边界条件崩溃）
- 性能问题（超时、吞吐下降、资源飙升）
- 集成问题（服务间调用/配置传播/权限/证书）

### 不适用场景
- 需求/验收标准不清导致的"看起来像 bug" → 先澄清需求（建议走 `/ease:flow-1-analyze-brd` 或补充用例/规则）
- 需要大规模设计决策/架构重构 → 先走架构评审（建议 `/ease:architecture-review`）

## 工作流程（强制）

你必须在进入下一个 Phase 之前完成当前 Phase，并产出对应“可审计输出”。

### Phase 1：根本原因调查（Root Cause Investigation）

**目标**：把“症状”变成“可复现的失败 + 初步定位范围”。

**执行步骤**：
1. **仔细阅读错误消息**
   - 把堆栈跟踪从头到尾读完，记录关键行号/文件路径/错误码
   - 优先围绕“报错的那一行 + 所在函数”理解问题，避免先通读大量无关代码
2. **一致地重现**
   - 给出精确重现步骤/命令（含参数、输入数据、环境变量）
   - 若不可重现：停止猜测，转为收集更多数据（日志级别、dump、请求样本）
3. **检查最近更改**
   - 代码改动：`git diff` / 最近提交
   - 配置/依赖/环境差异
4. **多组件系统收集证据（按边界打点）**
   - 在组件边界记录“进入数据/退出数据/关键配置/状态码”，跑一次得到证据，再据此缩小范围
5. **跟踪数据流（从失败点向上剥洋葱）**
   - 关注“错误输入/错误状态”最早出现的那一层
   - 需要系统化方法时，使用 `references/root-cause-tracing.md`

**产出物（必须给出）**：
- 可复现步骤（或不可复现说明 + 已收集证据）
- 失败证据：关键日志/堆栈/错误码/截图（文本化）
- 初步定位：怀疑模块/函数/边界层

**完成标准**（全部满足才算完成）：
- ✅ 能稳定复现，或已形成“不可复现的证据包”
- ✅ 已明确失败发生在哪个组件/边界层

### Phase 2：模式分析（Pattern Analysis）

**目标**：找到“正常工作”的参照物，并把差异显式列出来。

**执行步骤**：
1. **找到工作示例**：在同仓库找类似但正常工作的实现/测试/配置
2. **与参考比较**：逐行对照（不要略读），直到能解释差异
3. **识别差异**：列出每个差异，无论多小
4. **理解依赖与假设**：配置、环境、权限、网络、时序、数据形态

**产出物（必须给出）**：差异清单（建议用表格）

**完成标准**：
- ✅ 至少 1 个“工作示例”
- ✅ 差异清单可映射到可检验的假设

### Phase 3：假设与最小化实验（Hypothesis & Experiments）

**目标**：用最小实验验证单个假设，避免“改一堆试试看”。

**执行步骤**：
1. **形成单个可证伪假设**：
   - 格式："我认为 X 是根因，因为证据 Y；如果把变量 Z 从 A 改到 B，结果应当变为 R"
2. **设计最小实验**：一次只改一个变量
3. **运行并记录结果**：命令 + 输出摘要 + 结论（支持/否定）
4. **失败就回退**：不要在失败改动上叠加更多改动

**产出物（必须给出）**：
- 假设陈述
- 实验记录（建议直接用 `templates/debug-session-log.md`）

**完成标准**：
- ✅ 至少 1 个假设被明确“证实或证伪”

### Phase 4：实施最小修复（Implementation）

> ⚠️ **NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST**（能写测试就先写测试）。

**目标**：只修复根因，并用自动化验证证明它。

**执行步骤**：
1. **创建失败的测试用例/最小复现用例**
   - 不能直接加单测时：写一次性脚本或最小复现程序
2. **实施单个修复**：只覆盖根因，不捆绑重构
3. **验证修复**：
   - 原失败用例通过
   - 相关测试套件不回归
   - 真实重现路径消失
4. **3 次失败保护**：
   - 同类问题修复尝试达到 3 次仍失败：停止，回到 Phase 1，质疑架构/关键假设

**产出物（必须给出）**：
- 修复点说明（根因 -> 修复策略）
- 验证命令与结果摘要

**完成标准**：
- ✅ 原失败消失 + 回归通过

## 异常处理

### ❌ 无法复现

必须输出：
- 复现尝试矩阵（环境/数据/时间/依赖版本）
- 采样到的证据（日志/请求样本/监控指标）
- 下一步收敛策略（增加日志点/开启 debug/抓包/回放）

### ❌ 修复尝试达到 3 次仍失败

必须执行：
- 回到 Phase 1 重新收集证据
- 列出“已尝试修复 -> 预期 -> 实际结果”
- 质疑架构/假设（例如：共享状态、隐式耦合、错误边界）

## 何时使用

用于任何技术问题：
- 测试失败
- 生产中的bug
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**特别是在以下情况下使用：**
- 时间压力（紧急情况使猜测诱人）
- "只是一个快速修复"看起来明显
- 您已经尝试了多个修复
- 之前的修复不工作
- 您不完全理解问题

**即使在以下情况下也不要跳过：**
- 问题看起来简单（简单bug也有根本原因）
- 您在匆忙（匆忙保证重做）
- 经理希望现在修复（系统性比折腾更快）

## 四个阶段

您必须在进入下一个阶段之前完成每个阶段。

### 第1阶段：根本原因调查

**在尝试任何修复之前：**

1. **仔细阅读错误消息**
   - 不要跳过错误或警告
   - 它们通常包含精确解决方案
   - 把堆栈跟踪从头到尾读完
   - 记录行号、文件路径、错误代码
   - 优先围绕“报错的那一行 + 所在函数”理解问题，不要先去通读大量无关代码

2. **一致地重现**
   - 您能可靠地触发它吗？
   - 精确步骤是什么？
   - 每次都发生吗？
   - 如果不可重现 → 收集更多数据，不要猜测

   **网络/运行环境相关的补充规则（仅在重复验证之后）：**
   - 如果问题涉及外网访问（依赖/组件拉取、远程查询、第三方接口等），先在相同条件下明确“是否能稳定复现”
   - 只有在**多次重试仍然失败**、或表现为**超时/不稳定**时，才把“网络/运行环境（代理、DNS、证书、出口限制）”作为候选根因，并开始收集连通性证据

3. **检查最近的更改**
   - 什么更改可能导致这个？
   - Git diff、最近的提交
   - 新依赖、配置更改
   - 环境差异

4. **在多组件系统中收集证据**

   **当系统有多个组件（CI → 构建 → 签名，API → 服务 → 数据库）时：**

   **在提出修复之前，添加诊断工具：**
   ```
   对于每个组件边界：
     - 记录进入组件的数据
     - 记录退出组件的数据
     - 验证环境/配置传播
     - 检查每层的状态

   运行一次以收集显示它在哪里破坏的证据
   然后分析证据以识别失败的组件
   然后调查该特定组件
   ```

   **示例（多层系统）：**
   ```bash
   # 第1层：工作流
   echo "=== 工作流中可用的机密：==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # 第2层：构建脚本
   echo "=== 构建脚本中的环境变量：==="
   env | grep IDENTITY || echo "IDENTITY不在环境中"

   # 第3层：签名脚本
   echo "=== 钥匙串状态：==="
   security list-keychains
   security find-identity -v

   # 第4层：实际签名
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **这揭示：** 哪一层失败（机密 → 工作流 ✓，工作流 → 构建 ✗）

5. **跟踪数据流**

   **当错误在调用堆栈深处时：**

   参见此目录中的`root-cause-tracing.md`以获取完整的向后跟踪技术。

   **快速版本：**
   - 有问题的值/状态来自哪里？
   - 是谁把它传到这里的？
   - 继续向上追到“第一次变坏”的位置
   - 在源头修复，不在症状点修复

   **剥洋葱式读代码（避免读一大片无关代码）：**
   - 从异常信息/堆栈里给出的“文件 + 行号”切入，先只读该行所在的函数（以及它直接调用的少量分支）
   - 验证这个函数对输入的假设（空值、类型、范围、是否依赖环境变量/配置/网络）是否成立
   - 如果假设不成立，再沿堆栈向上一层，重复同样的“定位 → 阅读 → 验证”
   - 目标是用最小阅读量定位到“错误输入/错误状态”最早出现的那一层

### 第2阶段：模式分析

**在修复之前找到模式：**

1. **找到工作示例**
   - 在相同代码库中找到“类似但正常工作”的代码/测试/配置
   - 这些正常例子和当前问题点有哪些相同点？哪些不同点？

2. **与参考比较**
   - 如果实施模式，完全阅读参考实施
   - 不要略读 - 阅读每一行
   - 在应用之前完全理解模式

3. **识别差异**
   - 正常与异常之间有什么不同？
   - 列出每个差异，无论多小
   - 不要假设"那没关系"

4. **理解依赖**
   - 这需要什么其他组件？
   - 什么设置、配置、环境？
   - 它做出什么假设？

### 第3阶段：假设和测试

**科学方法：**

1. **形成单个假设**
   - 清楚说明："我认为X是根本原因因为Y"
   - 写下来
   - 具体，不是模糊

2. **最小化测试**
   - 用尽可能小的改动验证假设
   - 一次一个变量
   - 不要一次修复多个东西

3. **在继续之前验证**
   - 它工作了吗？是 → 第4阶段
   - 没工作？形成新假设
   - 不要在失败的改动上继续叠加更多改动

4. **当您不知道时**
   - 说"我不理解X"
   - 不要假装知道
   - 寻求帮助
   - 更多研究

### 第4阶段：实施

**修复根本原因，不是症状：**

1. **创建失败的测试用例**
   - 最简单的可能重现
   - 如果可能自动化测试
   - 如果没有框架一次性测试脚本
   - 在修复之前必须有
   - 使用`superpowers:test-driven-development`技能编写适当的失败测试

2. **实施单个修复**
   - 解决识别的根本原因
   - 一次一个更改
   - 没有"既然我在这里"的改进
   - 没有捆绑重构

3. **验证修复**
   - 测试现在通过？
   - 没有其他测试破坏？
   - 问题实际解决了？

4. **如果修复不工作**
   - 停止
   - 计数：您尝试了多少个修复？
   - 如果<3：返回第1阶段，用新信息重新分析
   - **如果≥3：停止并质疑架构（下面的步骤5）**
   - 不要在没有架构讨论的情况下尝试修复#4

5. **如果3+个修复失败：质疑架构**

   **指示架构问题的模式：**
   - 每个修复在不同地方揭示新的共享状态/耦合/问题
   - 修复需要"大规模重构"来实施
   - 每个修复在其他地方创建新症状

   **停止并质疑基础：**
   - 这个模式根本上是合理的吗？
   - 我们是"纯粹惯性地坚持它"吗？
   - 我们应该重构架构而不是继续修复症状？

   **在尝试更多修复之前与您的人类伙伴讨论**

   这不是一个失败的假设 - 这是一个错误的架构。

## 危险信号 - 停止并遵循过程

如果您捕捉到自己在想：
- "现在快速修复，稍后调查"
- "只是尝试更改X看看它是否工作"
- "添加多个更改，运行测试"
- "跳过测试，我会手动验证"
- "可能是X，让我修复它"
- "我不完全理解但这可能工作"
- "模式说X但我会不同地适应它"
- "这里是主要问题：[列出没有调查的修复]"
- 在跟踪数据流之前提出解决方案
- **"再一个修复尝试"（当已经尝试2+时）**
- **每个修复在不同地方揭示新问题**

**所有这些都意味着：停止。返回第1阶段。**

**如果3+个修复失败：** 质疑架构（参见第4.5阶段）

## 您的人类伙伴的信号您做错了

**注意这些重定向：**
- "那不是在发生吗？" - 您假设而没有验证
- "它会显示我们...？" - 您应该添加证据收集
- "停止猜测" - 您在不理解的情况下提出修复
- "深度思考这个" - 质疑基础，不只是症状
- "我们卡住了？"（沮丧） - 您的方法不工作

**当您看到这些时：** 停止。返回第1阶段。

## 常见合理化

| 借口 | 现实 |
|--------|---------|
| "问题简单，不需要过程" | 简单问题也有根本原因。对于简单bug过程快。 |
| "紧急，没有时间过程" | 系统性调试比猜测和检查折腾更快。 |
| "只是先试试这个，然后调查" | 第一次修复设定模式。从一开始就做对。 |
| "我会在确认修复工作后编写测试" | 未经测试的修复不持久。测试优先证明它。 |
| "一次多个修复节省时间" | 不能隔离什么工作。导致新bug。 |
| "参考太长，我会适应模式" | 部分理解保证bug。完全阅读它。 |
| "我看到问题，让我修复它" | 看到症状 ≠ 理解根本原因。 |
| "再一个修复尝试"（2+次失败后） | 3+次失败 = 架构问题。质疑模式，不要再次修复。 |

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|-------|---------------|------------------|
| **1. 根本原因** | 阅读错误、重现、检查更改、收集证据 | 理解什么和为什么 |
| **2. 模式** | 找到工作示例、比较 | 识别差异 |
| **3. 假设** | 形成理论、最小化测试 | 确认或新假设 |
| **4. 实施** | 创建测试、修复、验证 | Bug解决，测试通过 |

## 当过程揭示"没有根本原因"

如果系统性调查表明问题确实属于环境、时序依赖或外部因素：

1. 您已完成过程
2. 记录您调查的内容
3. 实施适当的处理（重试、超时、错误消息）
4. 添加监控/日志，便于后续快速定位

**但：** 95%的"没有根本原因"情况是不完整的调查。

## 现实世界影响

来自调试会话：
- 系统性方法：15-30分钟修复
- 随机修复方法：2-3小时折腾
- 首次修复率：95% vs 40%
- 引入的新bug：几乎为零 vs 常见
