# POC 执行流程

本文档详细描述 POC 验证的完整执行流程，确保验证过程规范、可重复。

## 流程概览

```
┌─────────────────────────────────────────────────────────────┐
│                    POC 执行流程                              │
├─────────────────────────────────────────────────────────────┤
│  [0] 系统分析 ─→ 读取项目文档，了解系统现状                    │
│        │                                                     │
│        ↓                                                     │
│  [1] 验证点分析 ─→ 提取、补充、细化验证点                      │
│        │                                                     │
│        ↓                                                     │
│  [2] 方案设计 ─→ 设计验证方案、用例、指标                      │
│        │                                                     │
│        ↓                                                     │
│  [3] 知识输出 ─→ 输出技术背景、最佳实践、问题延伸              │
│        │                                                     │
│        ↓                                                     │
│  [4] 代码实现 ─→ 生成可运行的验证代码                         │
│        │                                                     │
│        ↓                                                     │
│  [5] 验证执行 ─→ 运行代码、记录结果                           │
│        │                                                     │
│        ↓                                                     │
│  [6] 结论报告 ─→ 输出验证报告和建议                           │
└─────────────────────────────────────────────────────────────┘
```

## 阶段0: 系统分析

### 目标

充分了解项目现有系统架构、技术栈和约束，为 POC 验证提供上下文。

### 执行步骤

#### Step 0.1: 扫描项目结构

**优先使用专用工具**：
- 使用 `Glob` 查找文件（如 `Glob: CLAUDE.md, .claude/**/*`）
- 使用 `Grep` 搜索特定内容（如 `Grep: "pomxml|buildgradle|gomod|requirementstxt" in package.json`）
- 使用 `Read` 读取找到的文档内容

Shell 命令（如 `ls`）仅作为可选补充，当工具不足时使用。

#### Step 0.2: 读取核心文档

按优先级读取：

1. **P0 优先级**（必须读取）
   - `CLAUDE.md` 或 `.claude/` 目录
   - `docs/system/` 目录下所有文档

2. **P1 优先级**（应该读取）
   - `memory/constitution.md`
   - `README.md`

3. **P2 优先级**（建议读取）
   - 依赖配置文件
   - `docs/architecture/` 目录

#### Step 0.3: 提取关键信息

从文档中提取：

- 项目技术栈（语言、框架、版本）
- 现有中间件和外部依赖
- 架构风格和设计规范
- 技术约束和限制
- 与 POC 相关的现有组件

### 输出物

```markdown
## 系统现状分析

### 项目概述
- 项目名称：[名称]
- 项目类型：[Web应用/微服务/CLI工具/...]
- 主要功能：[简述]

### 技术栈
- 开发语言：[语言 + 版本]
- 主框架：[框架 + 版本]
- 构建工具：[Maven/Gradle/go mod/pip/...]

### 中间件
- 数据库：[类型 + 版本]
- 缓存：[类型 + 版本]
- 消息队列：[类型 + 版本]
- 其他：[...]

### 技术约束
- [约束1]
- [约束2]

### POC 相关组件
- [组件1]：[与 POC 的关系]
- [组件2]：[与 POC 的关系]
```

## 阶段1: 验证点分析

### 目标

从用户需求中提取完整的验证点清单，确保 POC 覆盖所有关键方面。

### 执行步骤

#### Step 1.1: 提取用户验证点

从用户描述中提取明确的验证需求：
- 用户明确提出的功能验证
- 用户关心的性能指标
- 用户担忧的风险点

#### Step 1.2: 补充隐含验证点

根据技术特性和系统现状补充：
- 技术原理相关的验证点
- 集成相关的验证点
- 运维相关的验证点

#### Step 1.3: 验证点分类

按六个维度分类验证点：

| 维度 | 验证内容 |
|------|----------|
| 功能性 | 核心功能是否正常工作 |
| 性能性 | 性能是否满足要求 |
| 可靠性 | 异常情况处理是否正确 |
| 兼容性 | 与现有系统是否兼容 |
| 可维护性 | 后续维护成本如何 |
| 安全性 | 是否存在安全风险 |

#### Step 1.4: 定义验证标准

为每个验证点定义明确的通过标准：
- 功能验证：预期行为描述
- 性能验证：量化指标阈值
- 可靠性验证：故障恢复预期

### 输出物

```markdown
## 验证点清单

### 用户验证需求
1. [验证点1]：[描述] - 标准：[通过标准]
2. [验证点2]：[描述] - 标准：[通过标准]

### 补充验证点
#### 功能性
- [F-001] [验证点]：[标准]

#### 性能性
- [P-001] [验证点]：[标准]

#### 可靠性
- [R-001] [验证点]：[标准]

#### 兼容性
- [C-001] [验证点]：[标准]

#### 可维护性
- [M-001] [验证点]：[标准]

#### 安全性
- [S-001] [验证点]：[标准]
```

## 阶段2: 方案设计

### 目标

设计可快速执行、可重复验证的 POC 方案。

### 执行步骤

#### Step 2.1: 确定验证环境

- 基于现有项目环境或独立环境
- 确定语言版本、依赖版本
- 确定外部服务依赖（数据库、缓存等）

#### Step 2.2: 设计验证用例

为每个验证点设计验证用例：

```markdown
| 用例ID | 验证点 | 前置条件 | 操作步骤 | 预期结果 | 验证方法 |
|--------|--------|----------|----------|----------|----------|
| TC-001 | F-001 | [条件] | [步骤] | [结果] | [方法] |
```

#### Step 2.3: 定义验证指标

```markdown
| 指标 | 目标值 | 测量方法 | 测量工具 |
|------|--------|----------|----------|
| 响应延迟 | < 100ms | P99 延迟 | wrk |
| 吞吐量 | > 1000 QPS | 并发测试 | k6 |
```

#### Step 2.4: 设计集成方案

说明如何与现有系统集成验证：
- 使用现有服务 Mock
- 对接现有接口
- 复用现有配置

### 输出物

完整的验证方案文档。

## 阶段3: 知识输出

### 目标

输出充分的技术知识，帮助用户全面理解被验证的技术。

### 执行步骤

#### Step 3.1: 技术背景

- 技术原理和工作机制
- 核心概念解释
- 架构和组件关系

#### Step 3.2: 最佳实践

- 官方推荐用法
- 社区最佳实践
- 常见使用模式

#### Step 3.3: 常见问题

- 常见错误和解决方案
- 性能陷阱
- 配置注意事项

#### Step 3.4: 问题延伸

- 生产环境注意事项
- 进阶使用场景
- 与其他技术的关联
- **与项目现有系统的潜在冲突**

#### Step 3.5: 参考资源

- 官方文档
- 推荐教程
- 相关项目

### 输出物

技术知识文档，作为验证报告的一部分。

---

## 阶段4: 代码实现

### 目标

生成可直接运行的验证代码。

### 执行步骤

#### Step 4.1: 创建项目结构

```
poc-[技术名称]/
├── README.md
├── [依赖配置]
├── src/
├── config/
└── docs/
```

#### Step 4.2: 编写核心验证代码

- 实现最小可验证的代码
- 添加充分的注释
- 确保结果可见可量化

#### Step 4.3: 编写验证用例

- 实现自动化验证脚本
- 输出结构化的验证结果

#### Step 4.4: 编写配置文件

- 提供默认配置
- 说明配置项含义

### 输出物

完整的验证代码项目。

---

## 阶段5: 验证执行

### 目标

执行验证代码，记录实际结果。

### 执行步骤

#### Step 5.1: 环境准备

```bash
# 检查依赖
# 安装组件
# 配置环境变量
```

#### Step 5.2: 执行验证

```bash
# 运行验证代码
# 记录输出
# 记录异常
```

#### Step 5.3: 收集结果

- 功能验证结果
- 性能测试数据
- 异常日志

#### Step 5.4: 结果分析

- 对比预期与实际
- 分析异常原因
- 评估验证结论

### 输出物

验证结果记录。

---

## 阶段6: 结论报告

### 目标

输出完整的 POC 验证报告。

### 报告内容

1. **验证摘要**
   - 验证目的
   - 验证结论
   - 关键发现

2. **系统现状**
   - 项目技术栈
   - 相关约束

3. **验证方案**
   - 验证点清单
   - 验证环境
   - 验证方法

4. **验证结果**
   - 功能验证结果
   - 性能验证结果
   - 集成验证结果

5. **技术知识**
   - 技术背景
   - 最佳实践
   - 常见问题

6. **结论与建议**
   - 综合评估
   - 推荐方案
   - 后续行动项

### 输出物

`docs/poc-report.md`
