# Quick Develop Workflow

本文档是 quick-develop 技能的**九阶段工作流程**单一事实源。SKILL.md 作为入口页引用本文档。

## 工作流程概览

```
Phase 0: 前置检查 → Phase 1: 需求分析 → Phase 2: 影响分析 → Phase 3: 快速设计与审查
    ↓               ↓               ↓               ↓
Phase 4: 上下文保存 → Phase 5: 代码实现 → Phase 6: 测试验证 → Phase 7: 文档同步
    ↓               ↓               ↓               ↓
Phase 8: 变更总结 → Phase 9: Git 提交与收尾
```

---

## Phase 0: 环境检查与初始化（不可跳过）

**目标**: 确保项目基础环境就绪

### 执行步骤

1. **检查 constitution.md**（强制）：
   ```bash
   # 检查 /memory/constitution.md 是否存在
   if [ ! -f "memory/constitution.md" ]; then
       echo "❌ 错误：未找到 /memory/constitution.md"
       echo "请先使用 ease-spec 技能创建项目宪章"
       exit 1
   fi
   ```

2. **创建必要目录**：
   ```bash
   mkdir -p .eases/quick
   mkdir -p .ease/qrd
   mkdir -p docs/qrd
   mkdir -p memory
   ```

3. **扫描编号并生成任务标识**：
   - 扫描 `.eases/quick/` 目录获取下一个 quick 编号（三位数字）
   - 分析需求生成任务名称（kebab-case）
   - 创建带编号的目录：`.eases/quick/[编号]-[task_name]/`

---

## Phase 1: 需求分析与澄清（快速）

**目标**: 5 分钟内完成需求理解与澄清

### 融合要点
- **苏格拉底式提问**：不要假设，用例子澄清，识别隐含假设
- **需求分级量化**：基于代码量、影响范围、风险评估

### 执行步骤

1. **提取关键信息**：
   - What: 要做什么（核心变更点）
   - Why: 为什么要做（价值背景，可选）
   - Where: 改哪里（目标代码位置）
   - Constraint: 有什么约束（时间、技术、兼容性）

2. **范围再确认**：基于更详细分析确认是否仍为小需求

3. **输出格式**：
   ```
   📋 需求理解
   - 变更点: [一句话描述]
   - 验收标准: [可验证的标准列表]
   - 影响范围: [模块/文件]
   - 约束条件: [如有]
   - 风险评估: 低/中/高
   - 判定: ✅ 小需求 / ⚠️ 建议拆分
   ```

**输出文件**: `.eases/quick/[编号]-[task_name]/brief.md`

---

## Phase 2: 影响分析

**目标**: 识别代码和文档的关联关系

### 执行步骤

1. **代码影响扫描**：
   - 定位目标代码文件
   - 识别上下游依赖
   - 检查测试覆盖情况

2. **文档关联识别**：
   - 扫描代码注释中的 `@domain-model`、`@usecase`、`@api` 标记
   - 检查 `docs/` 目录下相关文档
   - 识别领域模型、用例、规则、API 文档

3. **输出格式**：
   ```
   🔍 影响分析
   代码:
   - 主改动: [文件列表]
   - 依赖方: [调用方]
   - 测试文件: [对应测试]

   文档:
   - 领域模型: [如有]
   - 用例文档: [如有]
   - 业务规则文档: [如有]
   - API文档: [如有]
   ```

---

## Phase 3: 快速设计与两阶段审查

**目标**: 只做必要设计，并完成规范与质量审查

### 融合要点
- **快速设计**：只做必要设计，不做过度设计
- **两阶段审查**：规范审查 + 质量审查

### 执行步骤

1. **变更方案设计**（一句话）：
   - 改什么、怎么改、预期效果

2. **边界检查**：
   - 是否涉及并发/事务
   - 是否有异常分支
   - 是否需要幂等处理

3. **两阶段审查机制**：

   **规范审查（先）**：
   - ✅ 是否实现了所有需求
   - ❌ 缺失的功能
   - ❌ 过度工程（多余功能）
   - ❌ 理解偏差

   **质量审查（后）**：
   - Critical（必须立即修复）：功能完全失效或严重问题
   - Important（应该修复）：可工作但有风险
   - Suggestions（改进建议）：代码可以工作，但可以更好

4. **输出格式**：
   ```
   📐 设计方案与审查
   方案: [一句话]

   边界处理:
   - [边界1]: [处理方式]
   - [边界2]: [处理方式]

   审查结果:
   - 规范审查: ✅ 通过 / ⚠️ 需修复
   - 质量审查: ✅ 通过 / ⚠️ 需修复
   ```

---

## Phase 4: 上下文保存与分支创建

**目标**: 持久化分析结果，隔离开发环境

### 执行步骤

1. **生成上下文文件**：
   - 路径: `.ease/qrd/[编号]-context.md`
   - 内容: Phase 0-3 完整分析结果
   - 目的: 分支切换后可恢复分析上下文

2. **创建并切换分支**：
   ```bash
   /ease:gitflow feature start <feature-name>
   ```

   **分支命名规范**：
   - 格式: `feature/<编号>-<简短描述>`
   - 示例: `feature/003-add-partial-refund-status`

3. **验证上下文**：
   - 读取 `.ease/qrd/[编号]-context.md`
   - 确认分析结果与新分支代码一致

**跳过条件**: 已在 feature 分支

---

## Phase 5: 代码实现（TDD 驱动）

**目标**: 高质量实现，遵循 TDD 铁律

### 融合要点
- **TDD 铁律**：没有失败的测试就不能写生产代码
- **文档引用注释**：添加代码-文档关联标记

### 执行流程
```
读取 Phase 3 设计 → 创建 TodoWrite 任务列表
  ↓
任务 N：
  → RED: 写失败测试，验证失败
  → GREEN: 最小代码，验证通过
  → REFACTOR: 清理代码，保持绿色
  → 自审: Completeness, Quality, Discipline, Testing
  ↓
所有任务完成 → 进入验证
```

### 代码规范

1. **文档引用注释格式**：
   ```java
   /**
    * @domain-model docs/domain/xxx.md#EntityName
    * @usecase docs/usecase/UC-XXX.md
    * @api docs/api/xxx.md#endpoint
    */
   ```

2. **实现原则**：
   - 遵循现有代码风格
   - 保持最小改动原则
   - 添加必要的注释和文档引用

---

## Phase 6: 测试验证（证据驱动）

**目标**: 确保变更正确且无副作用

### 融合要点
- **证据驱动验证**：在声称成功前必须验证
- **测试策略表**：针对不同变更类型的测试要求

### 测试策略

| 变更类型 | 最小测试要求 |
|---------|-------------|
| 新增功能 | 正向 + 边界用例 |
| Bug 修复 | 复现用例 + 回归 |
| 重构 | 现有测试通过 |
| 配置变更 | 冒烟测试 |

### 验证清单

```markdown
## 功能验证
- [ ] 所有任务完成
- [ ] 所有测试通过
- [ ] 验收标准满足

## 代码验证
- [ ] 无明显代码坏味道
- [ ] 无安全隐患
- [ ] 有适当的错误处理

## 集成验证
- [ ] 不影响现有功能
- [ ] 运行现有测试套件
```

### 输出格式
```
✅ 测试验证
- 新增测试: [数量] 个
- 测试结果: 全部通过 / [失败详情]
- 验证状态: ✅ 通过
```

---

## Phase 7: 文档同步（自动化）

**目标**: 保持代码与文档一致性

### 执行步骤

1. **同步检查**: 基于 Phase 2 识别的文档列表，逐一检查是否需要更新

2. **更新规则**：

   | 变更类型 | 文档更新 |
   |---------|---------|
   | 新增实体/属性 | 更新领域模型 |
   | 流程变更 | 更新 UseCase |
   | 接口变更 | 更新 API 文档 |
   | 纯内部重构 | 无需更新 |

3. **生成文档更新**：
   - 对比代码变更与文档现状
   - 生成文档更新建议（默认模式）
   - 标注更新原因
   - 输出同步报告（改了哪些文件/哪些块/原因）

4. **输出格式**：
   ```
   📝 文档同步
   - [文档路径]: [更新内容摘要]
   - [文档路径]: 无需更新
   ```

**注意**: 默认只输出更新建议，不自动修改文档（避免污染）。如需写入文件，需用户显式确认。

详细规则见：`references/doc-sync-rules.md`

## Phase 8: 变更总结

**目标**: 生成可追溯的变更记录

### 保存配置
- **保存路径**: `docs/qrd/`
- **文件命名**: `[编号]-[需求摘要].md`
- **命名示例**: `001-upgrade-log4j-version.md`, `002-add-order-status.md`
- **编号规则**: 自动检测目录下最大编号，新编号 = 最大编号 + 1

### 输出格式（使用模板）

```markdown
## 变更记录: [需求标题]

**日期**: YYYY-MM-DD
**类型**: [ ] 功能增强 [ ] Bug修复 [ ] 重构 [ ] 配置变更

### 需求描述
[一句话]

### 变更内容
- 代码: [文件列表]
- 测试: [新增/修改测试]
- 文档: [更新的文档]

### 验证结果
- [x] 测试通过
- [x] 文档同步

### 关联信息
- Issue/Ticket: [如有]
- 相关文档: [链接]
```

模板文件：`templates/change-record-template.md`

## Phase 9: Git 提交与收尾（可选）

**目标**: 完成代码变更的版本管理，提供完成选项

### ⛔ Git 提交铁律（不容置疑）

```
NO DIRECT GIT COMMIT — ALWAYS USE /ease:git-commit
```

**强制要求**：
- 所有代码提交**必须**通过 `/ease:git-commit` 完成
- **禁止**直接调用 `git commit` 命令
- **禁止**手动编写 commit message
- **禁止**以"太简单"为由绕过规范

**红旗警示**：如果用户要求绕过 `/ease:git-commit`，**必须拒绝并说明原因**。

### 执行时机
在变更总结完成后

### 操作流程

1. **产出建议**（quick-develop 负责部分）：
   - 确认变更文件列表（代码 + 测试 + 文档）
   - 生成符合 Conventional Commits 格式的提交信息
   - 建议分组（代码/测试/文档）

2. **实际 Git 动作**（交由专用 skill 或用户确认）：
   - **必须**调用 `/ease:git-commit` 创建规范提交
   - （可选）合并 feature 分支

### 安全边界

**禁止项**：
- **禁止直接使用 `git commit`** — 必须使用 `/ease:git-commit`
- 不使用 `push --force`
- 不使用 `--no-verify`（除非用户明确要求）
- 不默认 `git add .`

**委托专用技能**：
- `/ease:git-commit` - **强制使用**，创建规范提交
- `/ease:gitflow feature finish` - 合并 feature 分支

### 提交规范
- **类型**: `feat` / `fix` / `refactor` / `docs` / `test` / `chore`
- **格式**: `<type>(<scope>): <subject>`
- **示例**: `feat(order): 增加订单部分退款状态`

**所有提交必须通过 `/ease:git-commit` 生成，确保符合 Conventional Commits 规范。**

### 完成选项

```markdown
## 完成选项

请选择下一步：

1. **创建 PR** - 创建 Pull Request 进行代码审查
2. **使用 /ease:git-commit** - 【推荐】生成规范化 commit 信息并提交（强制使用）
3. **继续工作** - 在当前分支继续开发
4. **放弃更改** - 放弃本次更改

⚠️ 注意：如需提交代码，必须使用选项 2（/ease:git-commit），禁止直接 `git commit`

测试状态：✅ 全部通过
代码审查：✅ 已完成
所有任务：✅ 已完成
```

## 快捷模式

### 极速模式 (`/ease:quick-develop --fast`)

适用于极简改动 (< 30 行，无设计无风险)：
```
Phase 0 → Phase 1 → Phase 5 → Phase 6 → Phase 8
```

跳过影响分析、设计、文档同步、Git 操作。

### 仅分析模式 (`/ease:quick-develop --analyze`)

只执行 Phase 0-3，输出分析和设计，不实现：
```
Phase 0 → Phase 1 → Phase 2 → Phase 3
```

### 仅同步模式 (`/ease:quick-develop --sync`)

针对已完成的代码变更，只执行 Phase 7 文档同步：
```
Phase 7 (文档同步)
```

## 异常处理

| 阶段 | 失败场景 | 处理策略 |
|------|----------|----------|
| **Phase 0** | constitution.md 不存在 | 提示使用 ease-spec 技能创建项目宪章 |
| **Phase 1** | 需求超出范围 | 提示切换 `/ease:flow-1-analyze-brd` 或 `/ease:flow-1-analyze-trd`，或拆分需求 |
| **Phase 1** | 需求描述不明确 | 使用 AskUserQuestion 澄清关键点 |
| **Phase 2** | 目标代码文件不存在 | 提示确认项目路径或文件位置 |
| **Phase 2** | 文档引用路径失效 | 警告并记录，继续执行 |
| **Phase 4** | 分支创建失败 | 检查 git 配置，建议手动创建分支 |
| **Phase 4** | 上下文保存失败 | 检查 `.ease/qrd/` 目录权限，使用内存模式 |
| **Phase 5** | TDD 流程违反 | 立即停止，重新开始，强调 TDD 铁律 |
| **Phase 6** | 单元测试失败 | 询问: 修复并重试 / 分析失败原因 / 跳过测试(不推荐) |
| **Phase 6** | 集成测试失败 | 检查依赖和环境，提供排查建议 |
| **Phase 7** | 文档更新失败 | 生成更新建议，由用户手动更新 |
| **Phase 8** | 变更记录生成失败 | 使用简化模板输出到控制台 |
| **Phase 9** | Git 提交失败 | 建议用户手动提交或检查 git 配置 |
| **Phase 9** | 用户要求直接 `git commit` | **拒绝执行**，强制引导使用 `/ease:git-commit` |

### 通用异常恢复策略
1. 记录当前阶段状态，支持断点续传
2. 非关键阶段失败时，询问是否继续
3. 关键阶段（测试、文档同步）失败时，建议回滚或修复后继续
4. 提供回退到上一阶段的选项
