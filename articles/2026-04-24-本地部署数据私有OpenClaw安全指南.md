# 本地部署、数据私有：OpenClaw 安全指南

> 你有没有想过，每次和 AI 聊天的内容都去了哪里？你的日程、邮件、工作文件——这些最私密的数据，真的安全吗？

当我第一次把 OpenClaw 连接到微信和邮箱时，身边朋友的第一反应不是"哇好酷"，而是："你的 AI 能看到你所有的聊天记录和邮件？这安全吗？"

好问题。今天就来聊聊 OpenClaw 的安全设计，以及你该如何配置，才能既享受 AI 助手的便利，又不用担心数据泄露。

## 为什么安全这件事很重要？

**先看几个现实场景：**

- 你让 AI 帮你整理邮件，它需要访问你的收件箱
- 你让 AI 帮你管理日程，它需要读取你的日历
- 你让 AI 帮你写代码，它会看到你的项目文件

这些数据一旦泄露，后果不堪设想。而市面上很多 AI 服务，你的数据是存在云端的，你根本不知道谁在后面看。

**OpenClaw 的核心安全理念：你的数据，你的规则。**

## OpenClaw 的安全架构

### 🏠 本地部署，数据不外泄

OpenClaw 最根本的安全优势：**它运行在你自己的机器上**。

```
传统 AI 服务：你的设备 → 云端服务器（数据在别人手里）
OpenClaw：    你的设备 → 本地 Gateway（数据在你自己手里）
```

这意味着：

- ✅ 你的聊天记录存储在你自己的机器上
- ✅ 你的配置文件只有你能访问
- ✅ 你的 API Key 保存在本地，不会上传到任何第三方
- ✅ 即使断网，大部分功能仍然可用

### 🔑 认证体系

OpenClaw 的认证设计采用**零信任**原则：

- 每个连接都需要独立认证
- 不存在"默认信任"的连接
- Token 有过期机制，不会永久有效
- 所有敏感操作都需要二次确认

### 🛡️ 权限分级

OpenClaw 对操作进行了严格的权限分级：

**🟢 低风险操作（自动执行）：**
- 读取文件
- 搜索网页
- 查看日历
- 内部计算和处理

**🟡 中风险操作（需要记录）：**
- 安装软件包
- Docker 容器操作
- 系统服务管理
- Cron 任务配置

**🔴 高风险操作（必须确认）：**
- 删除文件（特别是批量删除）
- 发送外部消息（邮件、推文）
- 修改认证配置
- 访问敏感目录

## 安全配置实操

### 1️⃣ 配置文件安全

OpenClaw 的核心配置文件在 `~/.openclaw/openclaw.json`，里面存储着你的 API Key 和各种凭证。

**保护措施：**
- 确保文件权限正确：`chmod 600 ~/.openclaw/openclaw.json`
- 不要把这个文件提交到 Git
- 定期检查文件内容，清理不再使用的 Key
- 使用环境变量替代明文存储敏感信息

**检查当前权限：**
```bash
ls -la ~/.openclaw/openclaw.json
# 应该显示：-rw------- (只有你自己能读写)
```

**如果权限不对，立即修复：**
```bash
chmod 600 ~/.openclaw/openclaw.json
```

### 2️⃣ API Key 管理

API Key 是 AI 助手的"钥匙"，管理不善后果严重。

**最佳实践：**
- 为 OpenClaw 创建专用的 API Key
- 设置使用额度上限
- 定期轮换 Key
- 不要在多个服务间共用同一个 Key
- 一旦怀疑泄露，立即撤销并重新生成

**配置示例（仅展示结构，不展示真实 Key）：**
```json
{
  "providers": {
    "openai": {
      "apiKey": "sk-YOUR_KEY_HERE"
    },
    "anthropic": {
      "apiKey": "sk-ant-YOUR_KEY_HERE"
    }
  }
}
```

### 3️⃣ 网络安全

**Gateway 访问控制：**

OpenClaw Gateway 是本地服务，默认只监听本机连接。但如果你需要远程访问，必须做好防护：

- 使用 HTTPS（不要用 HTTP）
- 配置强密码或 Token 认证
- 限制访问 IP 范围
- 使用 VPN 或 Tailscale 等安全隧道

**推荐的远程访问方案：**
```bash
# 方案1：Tailscale（推荐，零配置安全隧道）
# 安装 Tailscale 后自动获得加密连接

# 方案2：SSH 隧道（经典方案，稳定可靠）
ssh -L 7456:localhost:7456 your-server

# 方案3：Nginx 反向代理 + HTTPS（进阶方案）
# 配置 SSL 证书和访问控制
```

**不推荐的方案：**
- ❌ 直接暴露端口到公网
- ❌ 使用 HTTP 明文传输
- ❌ 简单密码认证
- ❌ 不限制 IP 访问

### 4️⃣ 消息平台安全

当你把 OpenClaw 接入微信、Discord、Telegram 等平台时，需要注意：

**通用安全建议：**
- 使用专用的 Bot 账号，不要绑定个人主账号
- 在群聊中，AI 只响应被 @ 的消息
- 设置敏感词过滤，防止 AI 泄露私人信息
- 定期审查 AI 的对话历史

**群聊安全配置示例：**
```json
{
  "channels": {
    "discord": {
      "respondTo": "mention_only",
      "maxResponseLength": 500,
      "forbiddenTopics": ["密码", "银行卡", "身份证"]
    }
  }
}
```

## 高级安全策略

### 🔄 审计日志

OpenClaw 记录了所有操作日志，定期检查可以帮助你发现异常：

**查看最近的操作记录：**
```bash
# 查看会话日志
ls -lt ~/.openclaw/sessions/

# 查看最近的文件操作
grep -r "write\|edit\|exec" ~/.openclaw/sessions/ | tail -20
```

**建议的审计频率：**
- 每周检查一次操作日志
- 关注文件写入和外部请求
- 特别注意深夜或异常时间的操作
- 发现问题及时调整权限配置

### 🧹 数据清理

定期清理不需要的数据，减少潜在风险：

```bash
# 清理旧的会话记录（保留最近30天）
find ~/.openclaw/sessions/ -mtime +30 -type f -delete

# 清理临时文件
rm -rf ~/.openclaw/tmp/*

# 清理旧的媒体文件
find ~/.openclaw/media/ -mtime +60 -type f -delete
```

**建议的清理策略：**
- 会话记录保留 30-90 天
- 临时文件用完即删
- 媒体文件按需保存
- 敏感文件单独加密存储

### 🔐 敏感操作防护

OpenClaw 内置了多层防护机制来处理敏感操作：

**文件操作防护：**
- 使用 `trash` 替代 `rm`，删除可恢复
- 批量删除前需要二次确认
- 敏感目录（如 `~/.ssh/`）有额外保护

**网络请求防护：**
- 拦截带有 Token/Key 的外发请求
- 禁止反弹 Shell
- 检测并阻止数据外泄行为

**认证防护：**
- 不允许 AI 修改自身的认证配置
- 关键配置文件有写保护
- 异常操作会触发告警

## 安全 Checklist

给你的 OpenClaw 做个安全体检：

**基础检查项：**
- ✅ 配置文件权限是否为 600
- ✅ API Key 是否定期轮换
- ✅ 是否使用了强密码或 Token
- ✅ Gateway 是否只监听本地或安全隧道
- ✅ 是否配置了操作审计

**进阶检查项：**
- ✅ 群聊中 AI 是否只响应 @ 消息
- ✅ 是否设置了敏感词过滤
- ✅ 是否定期清理旧数据
- ✅ 是否有数据备份机制
- ✅ 是否开启了 HTTPS

**终极检查项：**
- ✅ 是否审查过 AI 的所有外部请求
- ✅ 是否限制了 AI 的文件系统访问范围
- ✅ 是否有应急响应计划（Key 泄露怎么办）
- ✅ 是否定期更新 OpenClaw 到最新版本
- ✅ 是否在多台设备上使用不同的认证凭证

## 常见安全问题 FAQ

**Q：OpenClaw 会不会把我的数据发送到第三方？**
A：不会。OpenClaw 运行在你本地，只有调用 AI 模型 API 时才会发送必要的对话内容到模型提供商（如 OpenAI、Anthropic）。你可以选择使用本地模型（如 Ollama）来完全避免数据外发。

**Q：我的 API Key 安全吗？**
A：API Key 存储在本地配置文件中，只要你做好文件权限管理，就是安全的。建议设置 `chmod 600` 并定期轮换。

**Q：群聊里 AI 会不会泄露我的私人信息？**
A：OpenClaw 有多层防护。默认情况下，AI 在群聊中不会主动分享私聊内容。你可以通过配置 `respondTo: "mention_only"` 和敏感词过滤来进一步加强安全。

**Q：如果有人获得了我的 Gateway 访问权限怎么办？**
A：立即轮换所有 API Key，检查操作日志，重新生成 Gateway Token。建议从一开始就做好网络隔离和访问控制。

**Q：用 Docker 部署更安全吗？**
A：Docker 提供了一层额外的隔离，但安全性更多取决于你的配置。如果你不熟悉 Docker 安全配置，直接本机部署 + 好的权限管理可能更简单也更安全。

2. **这周完成**：设置群聊的响应规则和敏感词过滤

4. **长期坚持**：定期检查操作日志，保持 OpenClaw 更新

记住：**便利和安全不是对立的**。OpenClaw 的设计理念就是让你在享受 AI 助手便利的同时，始终掌控自己的数据。
