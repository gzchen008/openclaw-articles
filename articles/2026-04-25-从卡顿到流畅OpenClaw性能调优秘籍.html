AI 助手突然变得像蜗牛一样慢，一条消息等半分钟才回？或者 Mac mini 的风扇呼呼转，一看资源占用直接起飞？

别慌。今天我们就来聊聊 OpenClaw 的性能调优，从模型选择到系统配置，一步步帮你把响应速度拉满。

🎯 调优后你能得到什么

• 消息响应从 10 秒降到 3 秒以内

• 内存占用减少 40%

• 长对话不再越聊越慢

• 多任务并发也能稳如老狗

一、先找准瓶颈

性能问题千差万别，先搞清楚是哪个环节拖了后腿。

常见三大瓶颈：

1️⃣ 模型响应慢

最常见的。不同模型速度差异巨大——GPT-4 级别可能要 5-10 秒才出第一个字，轻量模型 1-2 秒就开始回复。

💡 判断方法：看 Gateway 日志里 ttft（Time To First Token）指标

2️⃣ 上下文太长

聊了 50 轮后突然变慢？大概率是上下文膨胀了。每条历史消息都要发给模型，token 越多，响应越慢。

💡 判断方法：日志里看 prompt_tokens，超过 8000 就该注意

3️⃣ 系统资源不足

老设备或同时跑了太多服务，内存和 CPU 成了瓶颈。

💡 判断方法：终端跑 top 看看 Node.js 进程的内存占用

二、模型选择：速度与质量的平衡术

这是调优投入产出比最高的一步。

日常对话用轻量模型

• 日常闲聊、简单问答用 turbo 级别模型

• 速度快 3-5 倍，成本更低

• 复杂场景（代码生成、长文写作）再切旗舰模型

💡 技巧：对话中用 /model 命令随时切换，简单问题用快模型，复杂问题切旗舰。

三、上下文管理：控制 Token 膨胀

上下文膨胀是"越聊越慢"的罪魁祸首。

策略一：设置合理的上下文窗口

限制只保留最近 N 个 token 的对话历史。默认值通常够用，但长对话变慢时可以适当缩小。

策略二：用总结代替原始历史

OpenClaw 的记忆系统天然支持——重要信息写入 MEMORY.md 和每日笔记，而不是全靠上下文窗口。

• 每隔 20-30 轮对话做一次小结写入 memory

• 新会话自动加载 memory，不堆砌历史

• 保留关键信息，token 数不失控

策略三：清理无用会话

定期清理过期会话，减少数据库负担。

四、系统层面优化

Node.js 内存限制

内存有限的设备（如树莓派），可以限制 Node.js 堆内存。512MB 对大多数场景足够。

日志级别调优

生产环境把日志级别设为 warn，减少磁盘 IO。调试时再改回 debug。

数据库维护

OpenClaw 使用 SQLite 存储会话数据。每月执行一次 vacuum，保持查询速度。

五、并发与资源调度

心跳间隔调优

默认 30 分钟一次心跳。根据使用频率调整——一天看几次消息的，45-60 分钟足够。

Cron 任务错峰

多个 Cron 任务不要挤在同一时间执行，错开几分钟，避免瞬间负载过高。

六、监控与诊断

定期检查这些指标，及时发现性能退化：

• Gateway 状态总览

• 最近请求的 ttft（首 token 延迟）

• 系统资源占用

• 磁盘空间（数据库膨胀预警）

• 网络延迟（影响模型 API 响应）

💡 把这些命令串起来做成 Cron 任务，每天自动跑一次，异常时发消息通知——这就是 OpenClaw 自我监控 的玩法。

七、极端情况：降级策略

有时候就是网络差、API 限流、服务器抽风。需要有降级方案：

模型自动降级

配置备用模型，主模型超时时自动切换到轻量模型。

本地模型兜底

如果部署了本地模型（如 Ollama），可以作为最后的兜底。网络全挂的情况下，至少还能用本地模型应急。

🎯 调优 Checklist

☐ 日常对话是否用了轻量模型？

☐ 上下文窗口是否设置合理（4000-8000）？

☐ 是否定期清理旧会话？

☐ 日志级别是否为 warn 或 info？

☐ 心跳间隔是否合理？

☐ Cron 任务是否错峰？

☐ 是否配置了 fallback 模型？

☐ 是否定期做数据库 vacuum？

写在最后

性能调优不是一次性的活，而是持续的过程。随着使用习惯变化、对话量增长、新功能加入，都需要定期回头看看有没有可以优化的地方。

最关键的一点：先测量，再优化。不要凭感觉调，用日志和指标说话。找到真正的瓶颈，针对性解决，效果立竿见影。
