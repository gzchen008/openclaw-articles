🤔 为什么需要命令审批？

用 OpenClaw 的朋友一定遇到过这个场景：AI 助手想帮你跑个命令，结果弹出一个审批提示，你还得手动点"允许"。一次两次还行，要是频繁操作，真的很烦！

特别是当你把 OpenClaw 当作全自动工作流引擎用的时候——定时任务、自动发布、批量脚本——每条命令都要审批，效率直接腰斩。

今天就来聊聊 OpenClaw 的命令审批机制，以及如何根据你的场景配置最合适的自动审批策略。✅

🔒 审批机制是什么？

OpenClaw 的命令审批（Exec Approvals）是一个安全护栏，防止 AI 助手在你不知情的情况下执行危险命令。

它有两层策略：

• OpenClaw 配置层（openclaw.json）：控制 exec 工具的行为策略

• 主机审批层（exec-approvals.json）：本地审批策略文件

两层策略取更严格的那个生效。所以只改一边可能不够，两边都要配对。

⚙️ 三个关键配置项

1️⃣ security（安全级别）

• deny：禁止所有主机命令执行

• allowlist：只允许白名单内的命令（推荐）

• full：允许一切（YOLO 模式）

2️⃣ ask（提示策略）

• off：永远不弹审批提示

• on-miss：白名单没匹配到时才提示（推荐）

• always：每条命令都提示

3️⃣ askFallback（无 UI 时的回退）

• deny：没有 UI 就拒绝（安全优先）

• allowlist：白名单匹配就放行

• full：全部放行

🚀 方案一：YOLO 模式（完全免审批）

适合个人开发环境，你完全信任 AI 助手，不想被任何审批打断。

第一步：配置 OpenClaw

1️⃣ 设置 exec 安全策略为 full

2️⃣ 关闭审批提示

3️⃣ 重启 Gateway 生效

命令行操作：

openclaw config set tools.exec.host gateway
openclaw config set tools.exec.security full
openclaw config set tools.exec.ask off
openclaw gateway restart

第二步：配置主机审批文件

编辑 ~/.openclaw/exec-approvals.json：

{
  "version": 1,
  "defaults": {
    "security": "full",
    "ask": "off",
    "askFallback": "full"
  }
}

或者用 CLI 一键设置：

openclaw approvals set --stdin <<'EOF'
{ version: 1, defaults: { security: "full", ask: "off", askFallback: "full" } }
EOF

⚡ 配置完成后，所有命令直接执行，不再弹审批。

🛡️ 方案二：白名单模式（推荐）

适合大多数场景，只信任常用命令，其他命令仍需审批。

配置文件 ~/.openclaw/exec-approvals.json：

{
  "version": 1,
  "defaults": {
    "security": "allowlist",
    "ask": "on-miss",
    "askFallback": "deny"
  },
  "agents": {
    "main": {
      "security": "allowlist",
      "ask": "on-miss",
      "askFallback": "deny",
      "autoAllowSkills": true,
      "allowlist": [
        { "pattern": "/usr/bin/git" },
        { "pattern": "/opt/homebrew/bin/gh" },
        { "pattern": "/opt/homebrew/bin/node" },
        { "pattern": "/usr/bin/python3" }
      ]
    }
  }
}

配置说明：

• allowlist：列出信任的可执行文件路径（支持 glob 通配符）

• autoAllowSkills: true：自动信任 Skills 引用的 CLI 工具

• on-miss：白名单没匹配到才会弹审批

这样 git、gh、node 等常用命令直接执行，其他命令仍需你手动审批。✅

📱 方案三：聊天审批转发

不想完全关闭审批，但也不想守在电脑前？可以把审批请求转发到你的聊天工具。

支持飞书、Telegram、Slack、Discord 等渠道。

配置示例（转发到飞书私聊）：

{
  "approvals": {
    "exec": {
      "enabled": true,
      "mode": "targets",
      "targets": [
        { "channel": "feishu", "to": "YOUR_USER_ID" }
      ]
    }
  }
}

配置后，审批请求会推送到飞书，你直接回复：

/approve <审批ID> allow-once    # 允许一次
/approve <审批ID> allow-always  # 永久加入白名单
/approve <审批ID> deny          # 拒绝

📱 随时随地审批，手机上就能搞定！

💡 Safe Bins：免审批的流式工具

OpenClaw 还内置了安全工具（Safe Bins）机制，以下命令在白名单模式下自动免审批：

• cut - 文本切割

• head / tail - 查看头尾

• tr - 字符替换

• wc - 统计计数

• uniq - 去重

这些工具只能处理 stdin 输入流，不能读写文件，所以是安全的。你还可以通过 tools.exec.safeBins 自定义添加。

⚡ 临时免审批（会话级）

如果你只是临时想让 AI 跑命令，不想改全局配置，可以用会话级快捷方式：

/exec security=full ask=off

或者更简单：

/elevated full

只在当前会话生效，关掉会话就恢复原样。适合临时调试场景。

🧹 常用运维命令

查看当前审批策略：

openclaw approvals get                  # 本地策略
openclaw approvals get --gateway         # Gateway 策略
openclaw approvals get --node my-mac     # 节点策略

设置策略：

openclaw approvals set --stdin < file.json    # 从文件设置
openclaw approvals set --node my-mac --stdin < file.json  # 设置节点策略

⚠️ 避坑指南

❌ 两边都要改：只改 openclaw.json 不改 exec-approvals.json 可能不生效，两层策略取更严格的

❌ 不要把解释器加到 safeBins：python3、bash、node 等不应该加入安全工具列表

✅ 生产环境用 allowlist：只信任必要的命令，其他走审批

✅ 开启 strictInlineEval：设置 tools.exec.strictInlineEval=true，防止 python -c 等内联执行绕过白名单

✅ 定期审查白名单：用 openclaw approvals get 检查是否有不该出现的命令

🎯 总结：怎么选？

🏠 个人开发机 → YOLO 模式（方案一），爽就完了

🏢 团队/生产环境 → 白名单模式（方案二），安全可控

📱 移动办公 → 聊天审批转发（方案三），随时掌控

🔧 临时调试 → 会话级快捷方式，用完即恢复

OpenClaw 的审批机制设计得很灵活，关键是根据你的信任边界选择合适的策略。个人用就放开，团队用就收紧，这才是正确的安全姿势。💪
