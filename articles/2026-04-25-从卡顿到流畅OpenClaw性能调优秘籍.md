## 一、先找准瓶颈

性能问题千差万别，先搞清楚是哪个环节拖了后腿。

**常见的三大瓶颈：**

**1️⃣ 模型响应慢**

这是最常见的。不同模型的速度差异巨大——GPT-4 级别的模型可能要 5-10 秒才出第一个字，而轻量模型 1-2 秒就开始回复。

**判断方法：** 看 Gateway 日志里 `ttft`（Time To First Token）这个指标。

**2️⃣ 上下文太长**

聊了 50 轮之后突然变慢？大概率是上下文膨胀了。每条历史消息都要发送给模型，token 数越多，响应越慢。

**判断方法：** 日志里看 `prompt_tokens` 数量，超过 8000 就该注意了。

**3️⃣ 系统资源不足**

老设备或者同时跑了太多服务，内存和 CPU 成了瓶颈。

**判断方法：** 终端跑 `top` 或活动监视器看看 Node.js 进程的内存占用。

## 二、模型选择：速度与质量的平衡术

这是调优投入产出比最高的一步。

### 日常对话用轻量模型

```json
{
  "agents": {
    "defaults": {
      "model": "zai/glm-5-turbo"
    }
  }
}
```

日常闲聊、简单问答、消息转发这类场景，用 turbo 级别的模型就够了，速度快 3-5 倍，成本也更低。

### 复杂任务用旗舰模型

需要深度思考的场景（代码生成、长文写作、复杂推理），再切换到旗舰模型：

```json
{
  "agents": {
    "defaults": {
      "model": "zai/glm-5"
    }
  }
}
```

**💡 技巧：** 你可以在对话中随时用 `/model` 命令切换模型，不用改配置文件。简单问题用快模型，复杂问题切旗舰，灵活调度。

## 三、上下文管理：控制 Token 膨胀

上下文膨胀是"越聊越慢"的罪魁祸首。几个实用策略：

### 设置合理的上下文窗口

```json
{
  "agents": {
    "defaults": {
      "contextWindow": 4000
    }
  }
}
```

限制上下文窗口大小，只保留最近 N 个 token 的对话历史。默认值通常够用，但如果你发现长对话变慢，适当缩小这个值。

### 用总结代替原始历史

OpenClaw 的记忆系统天然支持这个——重要的信息写入 MEMORY.md 和每日笔记，而不是全靠上下文窗口。

**最佳实践：**
- 每隔 20-30 轮对话，让 AI 做一次小结并写入 memory
- 新会话时自动加载 memory，而不是堆砌历史
- 这样既保留了关键信息，又不会让 token 数失控

### 清理无用会话

```bash
# 查看所有活跃会话
openclaw sessions list

# 清理不需要的会话
openclaw sessions prune --older-than 7d
```

## 四、系统层面优化

### Node.js 内存限制

如果你的设备内存有限（比如 4GB 的树莓派），可以限制 Node.js 的堆内存：

```bash
export NODE_OPTIONS="--max-old-space-size=512"
```

在 `~/.zshrc` 或启动脚本中加上这行，512MB 对大多数场景足够。

### Gateway 日志级别

生产环境把日志级别调高，减少磁盘 IO：

```json
{
  "log": {
    "level": "warn"
  }
}
```

调试时再改回 `debug`。日志写得少，系统负担就轻。

### 数据库维护

OpenClaw 使用 SQLite 存储会话数据。定期清理可以保持查询速度：

```bash
# 每月执行一次
openclaw db vacuum
```

## 五、并发与资源调度

### 心跳间隔调优

如果你开了心跳功能，默认 30 分钟一次。频率太高会浪费资源，太低又不够及时：

```json
{
  "heartbeat": {
    "intervalMinutes": 45
  }
}
```

根据自己的使用频率调整。一天看几次消息的，45-60 分钟足够。

### Cron 任务错峰

多个 Cron 任务不要挤在同一时间执行：

```bash
# ❌ 不好：全挤在整点
0 9 * * * task-a
0 9 * * * task-b

# ✅ 好：错开几分钟
0 9 * * * task-a
5 9 * * * task-b
```

## 六、监控与诊断

### 性能看板

定期检查这些指标，及时发现退化：

```bash
# Gateway 状态总览
openclaw gateway status

# 最近的请求延迟
openclaw gateway logs --tail 100 | grep ttft
```

### 常用诊断命令

```bash
# 查看系统资源
top -l 1 | grep -E "PhysMem|CPU"

# 查看磁盘空间（数据库膨胀会占空间）
df -h ~

# 查看网络延迟（影响模型 API 响应）
ping -c 5 api.openai.com
```

把这些命令串起来做成一个 Cron 任务，每天自动跑一次，异常时发消息通知你——这就是 OpenClaw 的自我监控玩法。

## 七、极端情况：降级策略

有时候就是网络差、API 限流、服务器抽风。这时候需要有降级方案：

### 模型自动降级

配置备用模型，主模型超时时自动切换：

```json
{
  "agents": {
    "defaults": {
      "model": "zai/glm-5",
      "fallbackModel": "zai/glm-5-turbo"
    }
  }
}
```

### 本地模型兜底

如果你有本地部署的模型（比如 Ollama），可以作为最后的兜底：

```json
{
  "providers": {
    "ollama": {
      "baseUrl": "http://localhost:11434"
    }
  }
}
```

网络全挂的情况下，至少还能用本地模型应急。
