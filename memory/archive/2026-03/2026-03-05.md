### 3. 技术实现模式
**CLI 封装模式**
- 适合：有官方 CLI 的工具（gh, notion-cli, linear-cli）
- 优点：快速实现，命令行友好
- 缺点：依赖 CLI 维护，高级功能可能缺失

**REST API 直接调用**
- 适合：没有 CLI 或需要高级功能
- 实现：通过 exec + curl 或专用 tool
- 优点：功能完整，控制精细
- 缺点：需要处理认证、限流、错误

**混合模式**
- CLI 处理常规操作（快速）
- API 处理高级功能（完整）
- 优点：兼顾效率与功能

### 4. Skill 设计原则
**渐进式加载**
- Metadata（name + description）：触发判断
- SKILL.md body：核心指导
- References/Scripts：按需加载

**资源复用**
- `scripts/`：可执行代码（确定性任务）
- `references/`：文档与模式（上下文补充）
- `assets/`：模板与资源（输出使用）

**错误处理**
- API 限流：指数退避 + 重试
- 认证失败：优雅降级 + 用户提示
- 网络超时：备用方案 + 状态缓存

### 5. 具体想法
**Notion Skill MVP**
- 核心功能：数据库查询、页面创建、搜索
- 技术栈：`notion-cli` + Notion API
- 参考实现：GitHub skill（CLI 封装模式）

**Calendar Skill 架构**
- 抽象层：统一 Calendar 接口（CRUD + 查询）
- 适配器：Google/Apple/Outlook 实现
- 技术栈：Google Calendar API + AppleScript

**Webhook 接收器**
- 目标：让外部服务推送事件到 OpenClaw
- 用例：GitHub PR 通知、Calendly 预约提醒
- 实现：Gateway 扩展或独立服务

## 技术笔记

### OpenClaw Skill 系统特点
- **模块化**：每个 Skill 独立目录，自包含
- **渐进式加载**：减少上下文占用
- **资源绑定**：scripts/references/assets 打包
- **触发机制**：通过 description 自动匹配

### 最佳实践
- SKILL.md 保持简洁（<500行）
- 详细文档放入 references/
- 可复用逻辑放入 scripts/
- 输出模板放入 assets/
